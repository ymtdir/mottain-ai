# Research: 食事カレンダー（食べた料理の記録・カレンダー閲覧）

Phase 0 の設計判断。既存スタック（001/002）を前提に、本機能で新たに決める点だけを扱う。Technical Context に未解決の NEEDS CLARIFICATION はない（スタックは継承）。

## R1. 承認の検知と記録の契機

- **Decision**: 承認検知はエージェントのツール呼び出しをゲートにする。`runAgent`（`src/server/agent/agent.ts`）の tools に `recordMealPlan` を1つ追加し、ユーザーが提案献立を承認したとモデルが判断したときにこのツールを呼ぶ。記録という副作用（`meal_logs` への INSERT）はツールの `execute` 内の決定的コードで行う。既存の `learnPreference` / `generateMealPlan` 等と同じ pattern。
- **Rationale**: 自由会話文の曖昧な自動判定で保存すると誤記録が入る（FR-002 / SC-006）。「ツールが呼ばれたときだけ記録」に閉じ込めれば、記録の契機が明示的になり制御が効く。エージェントに承認判断を委ねることは憲章 V（AIが価値の中心）にも合致し、新インフラも新規 ADR 系依存も不要（ADR-07 の範囲内）。
- **Alternatives considered**:
  - **明示的な確認 UI（「カレンダーに追加しますか？」ボタン）**: 誤検知は減るが操作が増え、「承認だけで反映（SC-002）」の体験を損なう。将来 UX 調整として足せる upgrade path。
  - **チャット完了後にサーバーが会話全体を解析して自動抽出**: 誤検知・二重記録の温床。不採用。
- **補足**: LLM の解釈確認（保存前の確認）は本 spec では設けない。誤記録は US4 の削除で正す。

## R2. 記録日の割当

- **Decision**: 承認日を起点に連続して割り当てる。献立の各 `Recipe` は `day: number`（1..N）を持つため、記録日 = 承認日 +（day − 1）で決定論的に算出する。月をまたぐ場合もそのまま連続日に記録する。基準日（「今日」）は Asia/Tokyo で一貫して扱う。
- **Rationale**: `Recipe.day` が既に 1..N のオフセットを持っており、追加のユーザー操作なしに承認日起点へ写像できる（SC-008）。単一ユーザー前提で TZ 固定にすればずれない。
- **Alternatives considered**:
  - **会話で日付を明示指定させる**（「金土日の献立」）: 柔軟だが承認フローが重くなる。承認日起点をデフォルトにし、明示指定対応は将来拡張。
  - **承認後にカレンダー UI 上でドラッグ配置**: MVP には過剰。不採用。

## R3. 食事記録のデータモデル（スナップショット vs 正準レシピ）

- **Decision**: `meal_logs` に「日付＋料理内容（title/ingredients/steps）」を自己完結のスナップショットで保持する。重複排除しない（同名でも別レコード）。002 の `saved_recipes` は変更せず、カレンダーは独立。将来お気に入りへ紐付ける場合に備え nullable な `sourceRecipeId` を足す余地は残すが、本 spec では持たない。
- **Rationale**: 「細かな違いのある同名料理は毎回別記録として残す（重複排除しない）」という仕様上、各記録は独立イベントであり、正準レシピを1行に集約する意味が薄い。スナップショットなら join も id 紐付けも不要で最も単純。002 のスキーマを触らないため、別ブランチで行うマイグレーション作業とも衝突しない。
- **Alternatives considered**:
  - **正準レシピ＋isFavorite フラグへの刷新**（全レシピを1つの表に保存し、カレンダーは recipeId 参照）: 設計会話で検討したが、(a) 002 スキーマの作り替えが必要、(b) dedup が「細かな違いは別記録」の方針と衝突、(c) レシピ編集が過去のカレンダーを書き換える副作用、のため不採用。
  - **meal_logs から recipeId で必ずお気に入りを参照**: 未お気に入りの料理（大多数）を記録できず不採用。
- **ADR**: この判断は `create-adr` で ADR 化する（実装前・R1 とまとめて可）。

## R4. カレンダーからのお気に入り登録

- **Decision**: カレンダー上の料理の「お気に入り登録」は、その料理内容（`meal_logs.content`）を 002 の `registerRecipe(content)`／`POST /api/recipes` にそのまま渡して行う。イラスト生成も 002 の既存フロー（お気に入り時のみ発火・非同期生成）を再利用する。新規エンドポイントは作らない。
- **Rationale**: `meal_logs` は 001 の `Recipe` と同形の内容を持つため、002 の登録フローに無改造で載る。重複排除・イラスト生成・状態表示はすべて 002 側で担保済み。FR-014（イラストはお気に入り時のみ）とも整合する。
- **Alternatives considered**: 記録とお気に入りを1つの表に統合: R3 で退けた正準レシピ化と同じ理由で不採用。

## R5. カレンダー UI（月グリッド）

- **Decision**: 月グリッドは自前コンポーネント（`MonthCalendar.tsx`）で実装し、各日セルにその日の料理名を表示する。新規依存は追加しない。前月・翌月の切替と当月データ取得（`GET /api/meals?month=YYYY-MM`）を行う。
- **Rationale**: shadcn/ui の Calendar（react-day-picker ベース）は日付選択用途で、日セルに任意コンテンツ（料理名の複数チップ）を並べる用途には合わない。月グリッド自体は曜日×週の単純な二次元描画で、Tailwind で十分軽量に作れる（YAGNI）。
- **Alternatives considered**:
  - **react-day-picker / 大型カレンダーライブラリ導入**: 内容セル表示にはオーバースペックで新規依存が増える。不採用。
  - **リスト（日付順）表示のみ**: 「その月をひと目で（SC-001）」の一覧性に劣る。月グリッドを主とし、必要なら日別詳細をダイアログで補う。

## R6. カレンダー画面のルーティングと導線

- **Decision**: チャットとは別ルート `/calendar`（`src/routes/calendar.tsx`）を新設する。チャット画面（`/`）とカレンダーを行き来する導線を既存サイドバー（`SessionSidebar`）もしくは共通ナビに追加する。
- **Rationale**: 「チャットとは独立した別画面（FR-007）」を満たす最小の方法。TanStack Router のファイルベースルートで完結し、新規依存不要。
- **Alternatives considered**: チャット内のモーダル/パネルとして出す: 「別画面」の要件と、月グリッドの表示領域確保の観点で劣る。不採用。
