# Quickstart: 食事カレンダー（食べた料理の記録・カレンダー閲覧）

本機能を end-to-end で検証する手順。詳細なモデル・エンドポイントは [data-model.md](./data-model.md)、設計判断は [research.md](./research.md) を参照。

## 前提

- 001（献立提案）・002（レシピ保存）が動作していること。
- ローカル DB に `meal_logs` テーブルが適用済みであること（マイグレーションの実適用は別ブランチ・最終フェーズ。ローカル検証時は生成済み migration を適用する）。
- 開発サーバ起動: `pnpm dev`（既存手順）。
- カレンダーはヘッダーの「カレンダー」タブで開く（`/calendar` ルートは廃止済み）。

## シナリオ 1: 承認した献立がカレンダーに記録される（US1・P1）

1. チャットで在庫と日数を伝え、3 日分の献立を提案させる。
2. 修正のやり取りをして、最後に「これでいい」「確定して」などと承認する。エージェントが「この献立を確定してよいですか？」と聞いてきた場合は「はい」と答える。
3. **期待**: エージェントが `recordMealPlan` を呼び、承認日・翌日・翌々日の連続 3 日に各日の料理が記録される。会話上は簡潔に「記録しました」。
4. 承認前（修正の途中）ではカレンダーに何も記録されないことを確認する（FR-002 / SC-006）。
5. 1 日に複数の料理を含む献立を承認 → 同じ日付に複数の料理が記録されることを確認する（FR-005）。

## シナリオ 1b: 同じ日に既存記録がある場合の上書き確認（fix・US1）

1. すでに記録がある日を含む献立を承認する（例: 今日の分がすでに記録済みの状態で今日から 3 日分の献立を確定する）。
2. **期待**: `recordMealPlan` が衝突を検出し、LLMが「X月Y日は既に「既存料理名」が記録されています。新しい「新料理名」に上書きしますか？」と確認する。
3. 「上書きして」と答えると、既存レコードが削除されて新しい料理が記録される。カレンダーで確認する。
4. 「そのままで」と答えると、その日はスキップされ、衝突しなかった日のみ記録される。
5. DB 制約の確認: 同じ `(userId, eatenOn)` で直接 INSERT しようとすると unique 制約違反でエラーになることを確認する（`0005_rapid_whistler.sql` 適用済みであること）。

## シナリオ 2: カレンダーで月内に食べた料理を見返す（US2・P1）

1. ヘッダーの「カレンダー」タブを開く。
2. **期待**: 当月の各日付マスに、その日記録された料理名が並ぶ（イラストなし＝FR-008）。
3. 料理名を選ぶ → 材料・手順がダイアログで表示される（FR-009 / SC-003）。
4. 前月・翌月に切り替える → その月の記録が表示される（FR-010）。月をまたいで記録された献立が各月で正しく見えることを確認する（Edge Case）。
5. 記録が 0 件の月を開く → エラーにならず空状態が表示される（FR-011）。

## シナリオ 3: カレンダーからお気に入り登録（US3・P2）

1. カレンダー上の料理の詳細ダイアログを開き「お気に入りに登録」する。
2. **期待**: 002 のお気に入りタブにそのレシピが現れ、再読み込み後も保持される（SC-009）。イラストは 002 の既存フローで生成される（FR-014）。
3. 同じ料理を再度お気に入り登録しても二重に増えないことを確認する（重複防止は 002 側で担保）。

## シナリオ 4: 誤って記録された料理を削除（US4・P3）

1. カレンダー上の料理の詳細ダイアログから「記録を削除」する。
2. **期待**: その料理がカレンダーから消える。
3. ページを再読み込みしても復活しないことを確認する（SC-007）。

## シナリオ 5: カレンダーからAIにコメントを送る（US5・P2）

1. カレンダー上の料理の詳細ダイアログを開き、コメント欄に「辛すぎた」「もう少し薄味がよかった」などと入力して「AIに送る」を押す。
2. **期待**: ボタンが「送信中...」になり、完了後に「AIに送りました。次回の献立に反映されます。」のトーストが表示される（FR-015/016）。チャットタブへの切り替えは起きない。
3. コメント欄が空の状態では「AIに送る」ボタンが押せないことを確認する（FR-017）。
4. 全角入力（IME）でEnterを押して変換確定してもコメントが送信されないことを確認する（IMEバグ修正）。
5. コメント送信後、サイドバーの「好みメモ」に反映された内容が表示されることを確認する（好みを言葉にしたコメントの場合のみ反映される）。チャットセッションは作成されない。

## 単体テスト（Vitest コロケーション）

`src/server/services/meal-log.test.ts` で決定的ロジックを検証する:

- `assignDates`: `day`(1..N) を承認日起点の `eatenOn` に写像する（月またぎ含む）。
- `monthRange`: `"YYYY-MM"` から `[月初, 翌月初)` を返す（当月抽出の境界・月末/年またぎ）。

LLM の実呼び出し（承認判断そのもの）・E2E は単体テストの対象外。
