# Feature Specification: 食事カレンダー（食べた料理の記録・カレンダー閲覧）

**Feature Branch**: `003-meal-calendar`

**Created**: 2026-07-06

**Status**: Draft

**Input**: チャットで提案・承認された献立を「その日に食べた料理」としてカレンダーに記録し（US1）、チャットとは別のカレンダー画面で、その月に何を食べたかを料理名の一覧としてひと目で見返し、選べば材料・手順まで確認できる（US2）ようにする。カレンダー上で良かった料理はお気に入り登録でき（US3）、誤って記録された料理は削除できる（US4）。001（献立コア）・002（レシピ保存）の続きとして「食べた記録の可視化」を担う spec。

## Clarifications

### Session 2026-07-06

- Q: カレンダーに料理が載る起点は何か（手動記録／お気に入りの登録日流用／提案の承認時） → A: チャットで提案された献立をユーザーが承認したタイミング。承認判定は LLM が行い、承認をはっきり検知したときだけ記録する（自由会話文からの曖昧な自動判定はしない）
- Q: カレンダーに載せる料理はお気に入りに限定するか → A: 限定しない。実際に食べた料理は未保存のものが多いため、承認された献立をそのまま記録する。厳密な重複排除はしない（細かな違いのある同名料理は別記録として毎回残してよい）
- Q: カレンダーにイラストを表示するか → A: 表示しない。カレンダー上は料理名のみ。イラスト生成のトリガーは従来どおりお気に入り登録時のみ（002 のまま）
- Q: 承認した献立の日付はどう決まるか → A: 承認した日を起点に today, +1, +2 … と連続で自動割当する（N 日分の献立なら承認日から連続 N 日）
- Q: 1 日あたりの粒度（朝昼夕の区分を持つか） → A: 区分は持たない。1 日に複数の料理を区分なしで記録できる
- Q: 記録とお気に入り（002）の関係 → A: 独立。002 の saved_recipes は変更しない。カレンダー記録は料理内容（名前・材料・手順）を自己完結でスナップショット保持する。将来お気に入りへの紐付けが要るなら nullable な参照を足す余地を残すが、本 spec では持たない
- Q: お気に入り登録の入口をカレンダーにも設けるか → A: 設ける。カレンダーの記録は料理内容を丸ごと持つため、そこから 002 の保存レシピとしてお気に入り登録できる（イラスト生成は 002 の既存フローで自動）。記録（meal_log）とお気に入り（saved_recipes）は別物のまま
- Q: 食べた料理への評価・フィードバックはどう扱うか → A: 本 spec の範囲外（次 spec 004）。「星評価」ではなく自由コメント（例:「味が薄かった」）を起点に好み学習を残す方式にする。コメントは料理名をキーに 001 の PreferenceProfile（recipeAdjustments）へ反映し、以降の提案に注入する（001 の既存機構をそのまま使う）。カレンダーはそのコメントの入口として使うが、コメント／学習データは meal_log ではなく料理名キーの好みプロファイルに紐づく

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 承認した献立が自動でカレンダーに記録される (Priority: P1)

ユーザーはチャットで献立の提案を受け、修正のやり取りを経て「これでいい」と承認する。承認すると、その献立の各料理が「その日に食べた（作る）料理」としてカレンダーに記録される。記録される日付は、承認した日を起点に連続して自動で割り当てられる（3 日分の献立なら承認日・翌日・翌々日）。記録には料理名・材料・手順が提案時と同じ内容で残るため、あとから何を食べたかを正確に振り返れる。

**Why this priority**: 「せっかく提案された数日分の献立も、チャットを閉じると何を食べたか覚えていない」という課題を解決する中核。記録が生まれて初めてカレンダーに意味が出るため、本機能の土台であり P1。

**Independent Test**: チャットで数日分の献立を提案・承認し、(a) 承認日を起点とする連続した日付に、(b) 各料理が名前・材料・手順つきで記録されることを確認できる。

**Acceptance Scenarios**:

1. **Given** チャットで N 日分の献立が提案されている, **When** ユーザーがその献立を承認する, **Then** 承認日を起点とする連続 N 日に各日の料理が記録される
2. **Given** 献立提案の途中で修正のやり取りをしている, **When** まだ承認していない, **Then** カレンダーには何も記録されない
3. **Given** 承認された献立, **When** 記録が作られる, **Then** 各料理の名前・材料・手順が提案時と同じ内容で保持される
4. **Given** 1 日に複数の料理を含む献立, **When** 承認する, **Then** その日付に複数の料理が区分なしで記録される

---

### User Story 2 - カレンダーで月内に食べた料理を見返す (Priority: P1)

ユーザーはチャットとは別のカレンダー画面を開き、月単位で「その月に何を食べたか」をひと目で見返せる。各日付のマスにはその日記録された料理名が並ぶ（イラストは表示しない）。料理名を選ぶと、その料理の材料と調理手順を提案時と同じ形式で確認できる。表示する月は前月・翌月に切り替えられる。

**Why this priority**: 記録（US1）を実際に使う出口。記録と閲覧が揃って初めて「食べたものを振り返る」体験が完成するため、US1 と同格の P1。

**Independent Test**: 記録が 1 件以上ある月でカレンダー画面を開き、(a) 日付ごとに料理名が表示され、(b) 料理を選ぶと材料・手順が見え、(c) 月を切り替えても破綻しないことを確認できる。

**Acceptance Scenarios**:

1. **Given** ある月に食べた料理が記録されている, **When** その月のカレンダーを開く, **Then** 各日付にその日の料理名が表示される
2. **Given** カレンダーに料理名が表示されている, **When** ある料理を選ぶ, **Then** その料理の材料と手順が表示される
3. **Given** カレンダーを表示中, **When** 前月／翌月に切り替える, **Then** その月の記録が表示される
4. **Given** 記録が 0 件の月, **When** その月を開く, **Then** エラーにならず空の状態が分かるように表示される
5. **Given** 1 日に複数の料理が記録されている, **When** その日を見る, **Then** 複数の料理名がすべて確認できる

---

### User Story 3 - カレンダーからお気に入り登録する (Priority: P2)

ユーザーはカレンダーで振り返って「これは良かった」と思った料理を、その場でお気に入りに登録できる。登録するとその料理は保存レシピ（002）として永続化され、お気に入り一覧から見返せるようになる。イラストも従来どおりお気に入り登録を起点に生成される。

**Why this priority**: 「食べた → 良かった → 保存」という自然な導線をカレンダー上で完結させる。記録・閲覧（P1）が成立していれば後付けできる価値なので P2。

**Independent Test**: カレンダー上の料理をお気に入り登録し、(a) お気に入り一覧に現れ、(b) 再読み込み後も保持され、(c) 同じ料理を重ねて登録しても二重に増えないことを確認できる。

**Acceptance Scenarios**:

1. **Given** カレンダーに料理が記録されている, **When** その料理をお気に入り登録する, **Then** その料理が保存レシピとして永続化され、お気に入り一覧に現れる
2. **Given** お気に入り登録した, **When** ページを再読み込みする, **Then** お気に入りは保持されている
3. **Given** すでにお気に入り登録済みの料理, **When** カレンダーから同じ料理を登録しようとする, **Then** 二重に増えず、登録済みとして扱われる

---

### User Story 4 - 誤って記録された料理を削除する (Priority: P3)

ユーザーは、実際には食べなかった料理や誤って記録された料理を、カレンダーから削除できる。削除した記録は一覧から消え、再読み込み後も復活しない。

**Why this priority**: 承認判定や日付割当の取りこぼしを人手で正す安全弁。記録・閲覧（P1）が成立していれば後付けできるため P3。

**Independent Test**: 記録済みの料理を削除し、カレンダーから消えること、再読み込み後も復活しないことを確認できる。

**Acceptance Scenarios**:

1. **Given** ある日に料理が記録されている, **When** その記録を削除する, **Then** カレンダーからその料理が消える
2. **Given** 記録を削除した, **When** ページを再読み込みする, **Then** 削除した記録は復活しない

---

### Edge Cases

- 承認された献立が月をまたぐとき（例: 月末に 3 日分を承認）→ 翌月にまたがって連続日に記録し、各月のカレンダーで正しく表示される
- 承認を誤検知して記録してしまったとき → 承認ゲートで誤検知自体を抑えつつ、残った誤記録は US4 の削除で正せる
- 同じ料理名を別の日（または同じ日）に複数回記録するとき → 重複排除せず、それぞれ独立した記録として残す
- カレンダーから同じ料理を重ねてお気に入り登録するとき → お気に入り（002）側は二重に増やさない（記録の重複とは扱いが異なる）
- 「今日」の基準（日付の割当・当月判定）→ 単一ユーザー前提で一貫した基準日を用い、ずれない
- 記録が 0 件の日／0 件の月 → 空状態を破綻なく表示する
- 承認された献立に日数・日付の手がかりが曖昧なとき → 承認日を起点に、献立に含まれる日数分だけ連続で割り当てる

## Requirements _(mandatory)_

### Functional Requirements

**記録（承認起点）**

- **FR-001**: System MUST チャットで提案された献立をユーザーが承認したと判断したとき、その献立の各料理をカレンダーに記録する
- **FR-002**: System MUST 承認をはっきり検知したときだけ記録し、提案・修正の途中や未承認の会話がカレンダーに記録されないようにする
- **FR-003**: System MUST 記録される各料理について、名前・材料・手順を提案時と同じ内容で保持する（自己完結のスナップショット）
- **FR-004**: System MUST 記録日を、承認日を起点に連続して自動割り当てする（N 日分の献立なら承認日から連続 N 日、月をまたいでもよい）
- **FR-005**: System MUST 1 日に複数の料理を、食事区分を持たずに記録できる
- **FR-006**: System MUST カレンダー記録を永続化し、セッションをまたいで保持する

**閲覧（カレンダー）**

- **FR-007**: System MUST 月単位のカレンダーを、チャットとは独立した別画面で表示する
- **FR-008**: System MUST 各日付に、その日記録された料理名を表示する（イラストはカレンダー上に表示しない）
- **FR-009**: Users MUST be able to カレンダー上の料理を選び、その料理の材料と手順を確認できる
- **FR-010**: Users MUST be able to 表示する月を前月・翌月に切り替えられる
- **FR-011**: System MUST 記録が 0 件の日・月でもエラーにならず、空状態を分かるように表示する

**お気に入り（002 との連携）**

- **FR-012**: Users MUST be able to カレンダー上の料理をお気に入りに登録でき、登録した料理は保存レシピ（002）として永続化される。同じ料理の重複登録は二重に増やさない

**削除**

- **FR-013**: Users MUST be able to 誤って記録された料理をカレンダーから削除でき、削除は永続化される

**イラスト（既存機能との境界）**

- **FR-014**: System MUST イラスト生成をカレンダー記録では行わず、従来どおりお気に入り登録時のみに限定する（002 の挙動を変えない。カレンダーからのお気に入り登録も同じ既存フローで生成する）

### Key Entities _(include if feature involves data)_

- **食事記録 (MealLog Entry)**: 「その日に食べた（作る）1 つの料理」。日付と、料理内容（名前・材料〔品目と分量〕・手順）のスナップショットを持つ。ユーザーに紐づき、永続化される。1 日に複数存在してよく、重複排除はしない。イラストは持たない。将来お気に入り（002 の保存レシピ）への参照を足す余地を残すが、本 spec では持たない
- **献立承認 (Meal Plan Approval)**: チャットで提案された献立に対するユーザーの承認という出来事。承認された献立（1〜N 日分の料理群）を食事記録へ落とす契機。承認判定は AI エージェントが行い、記録という副作用は承認をはっきり検知したときだけ確定的に実行される
- **お気に入り登録 (Favorite from Calendar)**: カレンダー上の食事記録を起点に、その料理内容を 002 の保存レシピとして登録する操作。保存レシピ（saved_recipes）は食事記録とは別物として管理され、重複は増やさない。イラスト生成は 002 の既存フロー（お気に入り時のみ）を再利用する

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: ユーザーは任意の月について、その月に食べた料理を 1 画面で見返せる
- **SC-002**: 献立を承認してからカレンダーに反映されるまで、追加の手操作を要しない（承認だけで記録される）
- **SC-003**: カレンダー上の任意の料理から、材料・手順に 1 アクションで到達できる
- **SC-004**: 記録された料理名・材料・手順は、提案時の内容と 100% 一致する（欠落・改変がない）
- **SC-005**: 記録は再読み込み後も 100% 保持される
- **SC-006**: 承認していない会話がカレンダーに記録されることは 0 件である（承認ゲートが効く）
- **SC-007**: 削除した記録は、再読み込み後も 100% 復活しない
- **SC-008**: N 日分の献立を承認したとき、承認日を起点とする連続 N 日に過不足なく記録される
- **SC-009**: カレンダー上の料理をお気に入り登録でき、再読み込み後も保持される（同じ料理を重ねて登録しても二重に増えない）

## Assumptions

- 対象は単一ユーザー／単一世帯（001・002 を踏襲）
- 「今日」および当月判定は、単一ユーザー前提で一貫した基準日（日本時間想定）を用いる
- 承認判定は AI エージェントが行い、記録という副作用は承認をはっきり検知したときだけ確定的に実行する（既存のエージェント／チャット基盤の延長。具体手段は plan／ADR で扱う）
- カレンダー記録は 002 のお気に入り（saved_recipes）とは独立し、002 のスキーマは変更しない。将来お気に入りへ紐付ける場合は nullable な参照を後から足す
- カレンダーからのお気に入り登録は 002 の保存レシピを作る既存フロー（重複排除・イラスト生成）を再利用し、meal_log と saved_recipes は独立のまま保つ
- 記録される料理内容は、001 の献立提案でレシピが持つ構造（料理名・材料〔品目と分量〕・手順）と同じ形式・粒度を指す
- 次 spec（004）で、食べた料理への自由コメント（例:「味が薄かった」）を起点にした好み学習を扱う。コメントは AI が実行可能な調整（例:「次回は〇〇を増やす」）に翻訳し、料理名をキーに 001 の PreferenceProfile（recipeAdjustments）へ残して以降の提案に注入する。コメントはカレンダー起点で入力し、チャットセッションを介さない単発・一方向の LLM 呼び出し（会話往復なし）で処理して会話履歴を汚さない。学習データは meal_log ではなく料理名キーの好みプロファイルに紐づく（004 で実装、本 spec は範囲外）
- 提案時に保存済み（特にお気に入り）と強く似た料理を検出してお気に入りを提示する、という提案側の味付けは本 spec の範囲外（将来の別 spec）。埋め込み検索・ベクトルストア等の新インフラは導入しない（憲章 I／将来の upgrade path として記録）
- カレンダー記録の実データ投入に必要なマイグレーションの実適用（db:migrate）は別ブランチで後ほど行うため本 spec の範囲外。スキーマ定義とマイグレーションファイルの生成までを範囲とする（002 の T003 と同じ扱い）
- 推定カロリー等の栄養情報は本 spec の範囲外
- Web アプリとして提供する。モバイル最適化は範囲外
