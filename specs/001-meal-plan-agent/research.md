# Research: 食材使い切り献立エージェント

**Date**: 2026-07-03 | **Branch**: `001-meal-plan-agent`

本書は技術選定の検討を「決定／理由／代替案」で残す。主要な選定は ADR に記録済み — インフラ（[ADR-04](../../docs/adr/04-infra-cloud-run.md)）、Web/フルスタックフレームワーク（[ADR-05](../../docs/adr/05-web-framework-tanstack-start.md)）、AI コア（[ADR-06](../../docs/adr/06-ai-core-gemini-enterprise-agent-platform.md)）、エージェント/オーケストレーション層（[ADR-07](../../docs/adr/07-agent-framework-vercel-ai-sdk.md)）。

---

## R1. エージェント/オーケストレーション層

- **決定**: **Vercel AI SDK（TypeScript）** を採用する（[ADR-07](../../docs/adr/07-agent-framework-vercel-ai-sdk.md)）。エージェントは「ツールを使う単一ループ」として TanStack Start のサーバー関数に実装。モデルは AI コア（Gemini Enterprise Agent Platform / Gemini、[ADR-06](../../docs/adr/06-ai-core-gemini-enterprise-agent-platform.md)）を呼ぶ。
- **理由**: ①モデルの使い分けが一級市民（解析＝軽量、計画＝上位、イラスト＝画像モデル。R6 のコスト抑制に直結）／②チャット UI/UX（`useChat`・ストリーミング・ツール途中表示・Generative UI）が最強で、献立/買い物リスト/根拠を動的に描画でき FR-018 に寄与／③成熟・TS ネイティブで TanStack Start（ADR-05）と一気通貫。使い切り計画・在庫更新・買い物リスト生成・味改善は AI SDK の「ツール」として表現する。
- **前提の両立**: Gemini Enterprise Agent Platform 経由で Gemini を使うため、当該プラットフォームを使う前提を満たす。
- **代替案**: Google ADK（TS/Python）＝ Agent Platform 純正でマルチエージェント・評価・Agent Runtime を備えるが、本件はモデル使い分けとチャット UX が主眼で AI SDK が上回り、TS 版は未成熟・Python 版は2言語2サービスで YAGNI に反する。マルチエージェント不要で強みが活きないため不採用（将来必要になれば後続 ADR で再検討）。／ Gemini API 直接＋自前オーケストレーション（ツール実行・ストリーミングを自作、遠回り）。

## R2. Web/フルスタックフレームワーク

- **決定**: **TanStack Start（TypeScript・React・Vite）**。UI とサーバー関数（BFF）を1アプリに置き、Docker 化して Cloud Run 単一サービスで動かす。詳細は ADR-05。
- **理由**: 高インタラクティブなチャット/SaaS 型に素直（クライアント優先）／ エンドツーエンド型安全でデータ契約の正しさ（ガードレール）に効く／ Vite で開発が速い／ Cloud Run（非 Vercel）に素の Node サーバとして乗る／ ストリーミングは Vercel AI SDK を併用。
- **代替案**: Next.js 16（成熟・大エコシステムだが RSC が高インタラクティブ画面で摩擦、強みの多くが Vercel 前提で Cloud Run では活きにくい）。比較の全文は ADR-05。

## R-Infra. 実行基盤・データストア 【ADR-04 で確定済み】

- **決定**: Cloud Run（実行）＋ Cloud SQL for PostgreSQL（永続化、JSON は JSONB）＋ Terraform（IaC）＋ GitHub Actions（CD）。
- **本機能での使い方**: リレーショナルに持つ実体（世帯・在庫・献立・レシピ・評価・保存）と、柔軟な構造（会話文脈・学習した好みプロファイル・根拠）を JSONB で持つハイブリッド。DB は `enable_cloud_sql = true` で有効化。

## R4. 好み・味付けの学習方式

- **決定**: 構造化した「好みプロファイル」を DB に保持し、提案時にプロンプトへ注入する軽量方式。評価（5段階）とコメント（味の感想）を集約し、ジャンル傾向と味付け傾向（塩加減・辛さ・こってり等の軸）を更新する。
- **理由**: ファインチューニングやベクトル検索基盤は個人ハッカソン規模に過剰（YAGNI）。プロファイル注入で「使うほど寄る」体験と根拠可視化（どの感想をどの調整に結びつけたか FR-035）を両立できる。
- **代替案**: ファインチューニング（コスト・運用過大）、埋め込み＋RAG（レシピ検索には有効だが MVP には過剰。将来の保存レシピ活用で再検討）。
- **相反する感想の扱い（Edge Case）**: 直近を重視しつつ、単発の振れに過反応しないよう平滑化（直近数件の傾向で調整）。

## R5. アレルギー/苦手の回避（ガードレール）

- **決定**: 二重化する。①プロンプト制約でエージェントに回避を指示、②**確定的（コード）フィルタ**で提案（献立・買い物リスト・イラスト）に回避対象が含まれないか最終チェックし、混入時はブロックして再生成/代替提示。
- **理由**: SC-001（混入0%）は LLM 出力の確率的性質だけでは担保できない。決定的チェックを最後段に置くことで「越えてはならない制約」を守る。TanStack Start の型安全と併せ、契約違反を早期に弾く。
- **代替案**: プロンプトのみ（混入リスクが残り不可）。

## R6. コスト暴走の抑制

- **決定**: ①イラストは確定/明示要求時のみ生成（FR-026）、②日数は1〜7日に制限（FR-031）、③既定は軽量な Gemini モデルを用い、必要時のみ上位モデル、④1リクエストあたりの LLM/画像呼び出し回数に上限、⑤修正ループ中の再生成を抑制。
- **理由**: constitution の軽さと、ハッカソン運用でのコスト事故防止。spec のガードレール意図（コスト暴走抑制）を運用/実装レベルで担保。
- **代替案**: 制御なし（コスト事故リスク、不可）。

## R7. レシピのイラスト生成

- **決定**: 画像生成モデル（Gemini の画像生成／Imagen 系）を用い、確定または明示要求時のみ発火。生成物はレシピに紐づけて保存し、修正のたびに再生成しない（FR-025/026）。
- **理由**: 無駄な再生成・コストを避けるユーザー意図（Clarify 済み）に合致。
- **代替案**: 毎回自動生成（コスト・体験ともに不利、不可）。

## R8. 会話・セッションの状態管理

- **決定**: Vercel AI SDK はリクエスト単位でステートレスなため、会話文脈（FR-023）と確定した在庫・献立・評価はいずれも Postgres に永続化し、リクエストごとに必要な文脈を読み込んでプロンプトへ渡す。会話の一時状態と永続データはテーブルで分離する。
- **理由**: 提案セッション内の文脈保持と、セッションをまたぐ永続化（FR-004 等）の要件を両立。ADK のセッション基盤に依存しないぶん、永続化を自前設計で単純化できる。
- **代替案**: すべてメモリ（永続要件を満たせない）、ADK のセッション/メモリサービスに委ねる（ADK 不採用のため対象外）。
