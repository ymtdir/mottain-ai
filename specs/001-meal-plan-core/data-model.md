# Data Model: 献立コア

spec の Key Entities を、永続化の観点で整理する。**永続**（DB）と**セッション内のみ**（非永続）を明確に区別する。

## 永続エンティティ（Cloud SQL / PostgreSQL）

### User（ユーザー）

MVP は単一ユーザー固定。回避・好みの紐づけ先。

| フィールド | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | ユーザー識別子（MVP は固定1件） |
| created_at | timestamptz | 作成日時 |

### DietaryConstraint（回避制約 / ハード）

アレルギー・苦手食材。**絶対的な除外条件**。提案・変更のたびにコンテキスト注入され、かつコード側の最終検証（R2）で突き合わせる。

| フィールド | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| user_id | uuid (FK→User) | |
| items | jsonb | 回避食材の配列（正規化名・別名・種別: allergy/dislike） |
| updated_at | timestamptz | 更新日時 |

- 不変条件 INV-1: 回避対象は献立・買い物リストに 0% 混入（SC-001）。
- 好みより常に優先（FR-023）。

### PreferenceProfile（好みメモリ / ソフト）

会話から学習した好みを保持する軽量な永続メモリ。**厳密なリレーションは張らず**、提案・変更時にまるごとエージェントのコンテキストへ注入する（research R8）。2つのレイヤーを持つ:

- **全体傾向**: 食材の好みに加え、辛さ・塩分・味の濃さ等の味の属性（例: 辛さ=控えめ、塩分=控えめ）
- **レシピ固有の調整**: 特定レシピに紐づく改善メモ（例: 麻婆豆腐→塩控えめ・にんにく追加）。レシピの同定・適用は LLM の推論に委ね、コード側で `recipe_id` の厳密一致は取らない

| フィールド | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| user_id | uuid (FK→User) | |
| memory | jsonb | 好みメモリ。全体傾向＋レシピ固有調整を緩い構造（属性・方向・対象レシピ名など）で保持。追記・更新される |
| updated_at | timestamptz | 更新日時。相反する感想は直近を優先（FR-023） |

> 設計メモ（research R8）: 回避（DietaryConstraint）は 0% 保証のため**構造化テーブル＋コード検証**にする一方、好み（本メモリ）は曖昧さを許容して**コンテキスト注入**で効かせる、という非対称な設計。両者は「永続化されたユーザー文脈」という兄弟だが、持ち方を意図的に分ける。`memory` を単一 JSONB にするか追記型メモ配列にするか等の具体スキーマは tasks で確定。

#### 味の感想の反映フロー（専用エンティティは持たない）

「少ししょっぱかった」「パンチが足りない」といった抽象的な味の感想は、**永続化する実体ではなく処理の流れ**として扱う: プロンプト（会話発話）→ LLM で解析し具体調整（塩分を控える／香辛料を足す等）に翻訳 → **PreferenceProfile.preferences を更新**。感想そのものを保存する専用テーブル（FlavorFeedback）は持たない（YAGNI）。トレースが必要になれば会話ログ（Conversation）で足りる。

## セッション内エンティティ（非永続）

会話状態としてのみ保持し、DB に永続化しない（research R3）。

### InventoryItem（在庫アイテム）

| フィールド | 型 | 説明 |
|---|---|---|
| name | string | 品目（正規化名） |
| quantity | string/number? | 概算数量（曖昧可: 「少し」「野菜いろいろ」） |
| perishability | enum | 一般的な日持ち区分（ドメイン知識で付与。個別消費期限は持たない） |

### MealPlan（献立）

| フィールド | 型 | 説明 |
|---|---|---|
| period_days | int (1–7) | 対象日数 |
| meals | Recipe[] | 日別の夕食（傷みやすい食材を早い日に配置） |

### Recipe / Meal（レシピ・食事）

| フィールド | 型 | 説明 |
|---|---|---|
| day | int | 何日目か |
| title | string | 料理名 |
| ingredients | {name, amount}[] | 必要食材と分量 |
| steps | string[] | 手順の要点 |
| notes | string? | 日持ち注意など（「早めに使う／冷凍を検討」） |

### ShoppingList（買い物リスト）

| フィールド | 型 | 説明 |
|---|---|---|
| items | {name, amount}[] | 献立に必要で手持ちにない食材のみ（不足分だけ） |

- 不変条件 INV-2: 買い物リストは手持ちで不足するぶんのみ（SC-002）。手持ちで足りる食材を含めない。

### Conversation（会話セッション）

在庫把握・提案・変更のやり取りの文脈。メッセージ列と、現在の在庫・直近の献立/買い物リストを保持する。永続化するかは tasks で判断（MVP は必須でない）。

## 関係と不変条件のまとめ

- User 1 — 1 DietaryConstraint、User 1 — 1 PreferenceProfile（永続）。味の感想は PreferenceProfile 更新のフローに畳み、専用エンティティを持たない。
- MealPlan・ShoppingList・InventoryItem・Conversation はセッション内で生成・消費（非永続）。
- INV-1: 回避対象は提案に 0% 混入（ハード、コード側最終検証で担保）。
- INV-2: 買い物リストは不足分のみ。
- INV-3: 回避（ハード）＞好み（ソフト）の優先順位を常に守る。
