# Research: 献立コア

技術スタックは ADR-04〜11 で確定済みのため、本 research は**設計判断**（アーキテクチャ上の未決事項）に絞る。

## R1. 回避・好みの表現と反映方法

- **Decision**: 回避制約（ハード）と好みプロファイル（ソフト）を単一の「ユーザー文脈ストア」として DB に永続化し、提案・変更時にエージェントのシステムコンテキスト（プロンプト）へ構造化して注入する。機械学習・埋め込み検索・専用学習パイプラインは用いない。
- **Rationale**: 「絶対に避ける」も「好みに寄せる」も、本質は『永続化されたユーザー文脈を提案に効かせる』こと。強さ（ハード/ソフト）が違うだけで機構は同じ。コンテキスト注入なら LLM の推論だけで両方を扱え、YAGNI（原則I）に合致。ハッカソン規模で最小の作り込みで実現できる。
- **Alternatives considered**:
  - ベクトル埋め込み＋類似検索で好みを学習: 規模に対して過剰。運用・コスト増（不採用）。
  - ルールエンジンで回避をコード側フィルタ: 回避はコード側の最終検証で二重化する価値があるが（下記 R4）、好みまでルール化するのは硬直的（好みは LLM 推論に委ねる）。

## R2. 回避（ハード制約）の 0% 保証

- **Decision**: 回避は「プロンプト注入」だけに頼らず、生成結果に対する**コード側の最終検証（ガード）**を併用する。献立・買い物リストに回避対象が含まれていないかを構造化データ上で検査し、混入時は再生成またはブロックする。
- **Rationale**: SC-001（混入0%）は越えてはならない制約。LLM 出力は確率的なので、決定的なコードチェックで担保する。回避対象は構造化（食材名の正規化・別名辞書）して突き合わせる。
- **Alternatives considered**: プロンプトのみで担保 → 0% を保証できない（不採用）。

## R3. 在庫の扱い（セッション内・非永続）

- **Decision**: 在庫は会話セッションの状態としてのみ保持し、DB に永続化しない。提案のたびにユーザーが申告した在庫を構造化して使う。
- **Rationale**: 毎週リセットする使い方が前提（spec の Assumptions）。永続化・差し引き・持ち越しのライフサイクルは不要で、持たないほうが単純（YAGNI）。
- **Alternatives considered**: 在庫を永続化し消費・持ち越しを追跡 → 前回 spec で複雑化した部分。今回は明確に不採用。

## R4. エージェントのツール設計（Vercel AI SDK）

- **Decision**: 単一エージェント＋ツールループ。ツールは概ね以下に分ける（詳細は tasks で確定）。
  - 在庫解釈（発話→構造化在庫）
  - 献立生成（在庫・日数・ユーザー文脈から N 日分を計画。使い切り最適化・日持ち考慮を内包）
  - 買い物リスト算出（献立と在庫の差分＝不足分のみ）
  - 献立変更（対象箇所の差し替えと買い物リスト再整合）
  - ユーザー文脈更新（回避・好みの登録/更新）
- **Rationale**: ツールで責務を分けると、回避検証（R2）・買い物リストの差分計算（SC-002）などをコード側の決定的ロジックに寄せられ、テスト（Vitest）しやすい。LLM は計画と自然言語解釈に集中。
- **Alternatives considered**: 全てを1つの巨大プロンプトで生成 → 検証・テストが困難（不採用）。

## R5. 画面構成

- **Decision**: チャット主体の単一画面。会話ログの中に、解釈した在庫の確認、献立（日別カード）、買い物リストを構造化して差し込む。回避・好み（永続のユーザー文脈）は設定領域から確認・編集できる。
- **Rationale**: 価値の中心がエージェントとの対話（原則V）。在庫・献立・買い物リストは会話の産物としてインラインに見せると自然。永続設定（回避・好み）だけは会話外からも参照・編集できると実用的。
- **Alternatives considered**: 在庫を常時表示する専用パネル → 在庫は非永続で毎回申告するため、常設パネルの価値が薄い（不採用）。

## R6. ユーザー識別（MVP）

- **Decision**: 単一ユーザー固定（認証なし）。DB 上は固定のユーザー識別子1件に、回避・好みを紐づける。
- **Rationale**: 単一ユーザー／単一世帯が前提（spec）。認証・アカウント基盤は YAGNI。将来必要になれば後続 ADR で導入。
- **Alternatives considered**: 最初から認証導入 → 過剰（不採用）。

## R8. 永続化戦略 — コンテキストで持つか、テーブルで持つか（設計の核）

- **Decision**: 「何でもテーブルに正規化する」のではなく、情報ごとに**コンテキスト注入で持つか／構造化テーブルで持つか**を取捨選択する。エージェントが価値の中心（原則V）である以上、曖昧・発展的な知識は自然言語に近い形でコンテキストに持たせ、正確性・決定性が要る情報だけをテーブルにする。判断軸は次のとおり。

  | 情報 | 持ち方 | 理由 |
  |---|---|---|
  | アレルギー・苦手食材（回避） | **テーブル**（構造化） | 0% 混入を守るためコード側の決定的検証（R2）が要る。突き合わせに構造が必要 |
  | 全体的な味の好み傾向 | **コンテキスト**（軽量な永続メモリを注入） | 曖昧で LLM 推論に委ねてよい。厳密なスキーマは不要 |
  | レシピ固有の改善（例: この麻婆豆腐は塩控えめ・にんにく追加） | **コンテキスト**（同上のメモリに追記） | レシピ同定も適用も LLM の推論が自然。硬いテーブルにすると別名・類似レシピの揺れに弱い |
  | 在庫 | **持たない**（セッション内のみ） | 毎週申告する前提（R3） |

- **Rationale**: レシピ固有の改善（FR-020）を厳密な `recipe_id → 調整` テーブルにすると、「麻婆豆腐」「マーボー豆腐」「麻婆豆腐（辛さ控えめ版）」といった同定の揺れを人手で吸収する羽目になり、YAGNI に反する。エージェントに「これまでの好み・レシピ改善メモ」をコンテキストとして渡せば、同定と適用を推論で扱える。一方、回避だけはコンテキスト任せにすると 0% を保証できないためテーブル＋コード検証にする、という**非対称**が要点。
- **具体化**: 好み（全体傾向＋レシピ固有調整）は、単一の永続「ユーザーメモリ」（DietaryConstraint とは別の、緩い JSONB もしくは自然言語に近い構造）として保持し、提案・変更時にまるごとエージェントのコンテキストへ注入する。厳密なリレーションは張らない。
- **Alternatives considered**:
  - すべてをテーブルに正規化（recipe テーブル・adjustment テーブル等）: 同定・整合の作り込みが増え、ハッカソン規模で過剰（不採用）。
  - すべてをコンテキストに寄せる（回避もプロンプトのみ）: 回避 0% を保証できない（不採用）。
- **未決（tasks で確定）**: 「ユーザーメモリ」を JSONB カラム1つにするか、追記型のメモ配列にするか等の具体スキーマ。注入時のトークン量が増えた場合の要約・剪定の要否。

## R7. 段階的実装の順序

- **Decision**: ①チャット疎通（LLM 往復の土台）→ ②在庫解釈 → ③献立・買い物リスト生成（＋回避の最終検証）→ ④対話変更 → ⑤好み学習（ユーザー文脈注入）。
- **Rationale**: 「まずチャットが動く」土台を最初に通し、以降は US1→US2→US3→US4 の価値順に積む。各段が独立して確認可能。
- **Alternatives considered**: 全機能一括実装 → 検証点が遅れる（不採用）。
