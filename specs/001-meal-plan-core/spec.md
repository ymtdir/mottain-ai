# Feature Specification: 献立コア（在庫→献立・買い物リスト＋回避＋対話変更＋好み学習）

**Feature Branch**: `001-meal-plan-core`

**Created**: 2026-07-03

**Status**: Draft

**Input**: チャットで在庫を伝えて N 日分の夕食献立と買い物リストを受け取り（US1）、アレルギー・苦手食材を絶対回避しつつ（US2）、対話の中で献立を変更でき（US3）、その際に述べた好みを学習して以降の提案へ反映する（US4）。5段階評価インターフェースは本 spec に含めず、同じ好み学習の核に後続 spec で足す。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - チャットで在庫を伝えて N 日分の献立と買い物リストを受け取る (Priority: P1)

ユーザーはチャットで「冷蔵庫にじゃがいもと人参、玉ねぎがある。3日分の夕食を考えて」のように自然に話す。エージェントは発話から在庫を読み取り、手持ち食材を無駄なく使い切ることを優先した N 日分（1〜7日）の夕食献立と、手持ちで足りないぶんだけの買い物リストを返す。在庫は毎回チャットで申告する前提とし、在庫量をセッションをまたいで記憶・持ち越しする必要はない（使い切れなかった余りは次回の申告で伝えればよい）。

**Why this priority**: これがプロダクトの中核体験であり、単独で「今日何作ろう／これ使い切らなきゃ」という課題を解決する最小の価値提供になる。これだけでも動く MVP になる。

**Independent Test**: 在庫と日数をチャットで伝え、(a) 指定日数分の献立が返り、(b) 買い物リストが手持ちで不足するぶんだけで構成され、(c) 手持ち食材が献立全体で使われていることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが手持ち食材と「3日分」を伝えた, **When** 献立提案を要求する, **Then** 3日分の夕食献立と、手持ちで不足するぶんだけの買い物リストが提示される
2. **Given** 手持ちに複数の生鮮食材がある, **When** 献立が提案される, **Then** それらの食材が期間内の献立で使い切られる（余りを最小化する）構成になっている
3. **Given** 手持ちだけで全日程を賄える, **When** 献立が提案される, **Then** 買い物リストは空（または「買い足し不要」）になる
4. **Given** 傷みやすい食材と日持ちする食材が混在する, **When** 献立が提案される, **Then** 傷みやすい食材がより早い日に配置される

---

### User Story 2 - アレルギー・苦手食材を絶対に避ける (Priority: P1)

ユーザーはアレルギーや苦手な食材（例: 甲殻類アレルギー、ピーマンが苦手）を登録できる。エージェントは登録された食材を、いかなる献立・買い物リストにも絶対に含めない。回避すると献立が成立しない場合は、混入させる代わりにその旨と代替案を伝える。

**Why this priority**: 安全に関わる越えてはならない制約（ガードレール）。中核体験と同格の P1 とし、他のどの機能よりも優先して守られなければならない。

**Independent Test**: アレルギー・苦手食材を登録した状態で複数回の献立提案を行い、提案（献立・買い物リスト）に登録食材が一切含まれないことを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが特定食材をアレルギー/苦手として登録している, **When** 献立が提案される, **Then** その食材は献立にも買い物リストにも一切含まれない
2. **Given** 手持ち食材に登録済みの回避対象が含まれる, **When** 献立が提案される, **Then** その食材は使い切り対象から除外され、献立に用いられない
3. **Given** 回避対象を除くと要求どおりの献立が組めない, **When** 献立を要求する, **Then** エージェントは回避対象を混入させず、成立しない旨と代替案を伝える
4. **Given** ユーザーがアレルギー・苦手食材を追加/更新する, **When** 次の献立を要求する, **Then** 更新後の回避対象が尊重される

---

### User Story 3 - 対話で献立を変更する (Priority: P2)

提案された献立が気に入らない場合、ユーザーはチャットで「この日を魚料理に」「もっと簡単に」「麻婆豆腐は辛いから別のものに変えて」などと変更を依頼できる。エージェントは該当箇所を変更し、買い物リストも矛盾なく更新する。変更対象以外は不必要に崩さない。変更理由として好み（例: 辛いのが苦手）が述べられた場合は、それを US4 の好み学習に引き渡す。

**Why this priority**: 提案の当たり外れを吸収し満足度を高める。初回提案（P1）が成立していれば後付けできるため P2。

**Independent Test**: 提案に対し変更を依頼し、該当箇所が変更され、買い物リストが更新後の献立と整合すること、変更対象以外が不必要に変わらないことを確認できる。

**Acceptance Scenarios**:

1. **Given** 献立が提案されている, **When** 特定の日や条件の変更を依頼する, **Then** 該当部分が変更され、それ以外は不必要に崩れない
2. **Given** 変更で必要な食材が変わった, **When** 変更が適用される, **Then** 買い物リストが更新後の献立と整合する（過不足がない）
3. **Given** ユーザーが「麻婆豆腐は辛いから変えて」のように理由付きで変更を依頼する, **When** 変更が適用される, **Then** 該当が別レシピに変わり、述べられた好み（辛さが苦手）が以降の提案に反映される

---

### User Story 4 - 会話から好みを学習して反映する (Priority: P3)

ユーザーが会話の中で述べた好み・味の感想（食材の好き嫌いに加え、「辛いのが苦手」「パンチが足りない」「少ししょっぱかった」といった抽象的・感覚的な味の属性も含む）を、エージェントが捕捉する。抽象的な感想は具体的な調整（例: 辛さを抑える、香辛料を足す、塩分を控える）に翻訳し、以降の献立提案に反映する。学習した好みは永続化され、使うほどユーザーの好みに寄っていく。本 spec では入口を「会話でのフィードバック」に限定し、5段階評価による入口は後続 spec で同じ核に足す。

**Why this priority**: 継続利用で価値が増す差別化要素。抽象的な感覚しか言えないユーザーも、会話から具体的な改善を受け取れる。初期の単発提案には必須でないため P3。

**Independent Test**: 会話で好み・味の感想を伝え、(a) 抽象的な感想に対し具体的な調整が次回提案に反映され、(b) その好みが永続化されて別セッションでも反映されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが会話で「辛いのが苦手」と伝える, **When** 次の献立が提案される, **Then** 辛い料理を避ける／辛さを抑えた提案になる（全体傾向の反映）
2. **Given** ユーザーが特定レシピに「少ししょっぱかった」と伝える, **When** 次回そのレシピ（同じ／類似）が提案・生成される, **Then** そのレシピの塩分を控える具体調整が自動で適用される（レシピ固有の改善）
3. **Given** ユーザーが「パンチが足りない」と伝える, **When** 改善が提案される, **Then** 香辛料・にんにく等を足すといった具体的な手段が反映される
4. **Given** 学習した好みが蓄積されている, **When** 別セッションで献立を要求する, **Then** 蓄積した好み（全体傾向・レシピ固有調整の両方）が反映される（永続）
5. **Given** 好みの蓄積がまだ少ない, **When** 献立が提案される, **Then** 学習不足でも破綻せず、無難な提案が返る

---

### Edge Cases

- 在庫がほとんど無い／空の状態で献立を要求したとき（買い物リスト中心の提案に切り替わるか）
- 指定日数が範囲外のとき（0日以下、または上限の7日を超える）→ 上限内に収めるよう案内する
- 手持ちだけでは指定日数を賄えないとき（不足分は買い物リストで補う）
- 回避対象（アレルギー・苦手）を除くと現実的な献立が組めないとき（混入させず、成立しない旨と代替案を伝える）
- 手持ち食材の数量が曖昧・不明なとき（「野菜いろいろ」「少し」）
- 一般的な日持ちを超えて使う提案になる食材があるとき（「早めに使う／冷凍などを検討」と短く注意を添える）
- 使い切り計画とアレルギー回避が競合するとき（回避が最優先される）
- 好みの反映とアレルギー回避が競合するとき（回避が最優先される）
- 相反する要望が同時に出たとき（「簡単に」かつ「手の込んだ料理を」）
- 変更を短時間に何度も依頼するとき（毎回、買い物リストを整合させる）
- 味の感想が過去と相反するとき（前回「しょっぱい」→今回「味が薄い」など）、直近の感想を優先する

## Requirements _(mandatory)_

### Functional Requirements

**在庫の把握（チャット）**

- **FR-001**: System MUST ユーザーが手持ちの食材・調味料を自由な発話で伝えられるようにし、分類入力を強制しない
- **FR-002**: System MUST 発話を解釈し、品目と（推定できる場合は）概算数量からなる在庫として構造化する
- **FR-003**: Users MUST be able to 解釈された在庫をチャットで確認・修正できる（追加・削除・数量調整）
- **FR-004**: System MUST 在庫を当該セッション（今回の献立考案）内で保持・利用する。在庫量をセッションをまたいで永続化・持ち越ししない（次回はユーザーが再申告する）

**献立と買い物リスト**

- **FR-005**: Users MUST be able to 欲しい献立の日数を 1〜7 日の範囲でリクエストごとに指定できる（例: 3日分、今週分）
- **FR-006**: System MUST 範囲外の日数（0日以下、または7日超）を指定された場合、上限内に収めるよう案内する
- **FR-007**: System MUST 指定された日数分の夕食献立を生成する
- **FR-008**: System MUST 献立全体を通して手持ち食材の使い切りを優先し、余り（廃棄）を最小化する
- **FR-009**: System MUST 手持ちで不足するぶんだけを含む買い物リストを生成する（手持ちで足りる食材を含めない）

**日持ち（鮮度）の考慮**

- **FR-010**: System MUST 食材ごとの一般的な日持ちを考慮し、傷みやすい食材を献立のより早い日に配置する（在庫ごとの消費期限は個別管理しない）
- **FR-011**: System MUST 一般的な日持ちを超えて使う提案になる食材について「早めに使う」または「冷凍など保存を検討」といった短い注意を添える（保存方法の判断はユーザーに委ねる）

**回避制約（ガードレール）**

- **FR-012**: Users MUST be able to アレルギー・苦手食材を登録できる
- **FR-013**: System MUST NOT 登録されたアレルギー・苦手食材を、いかなる献立・買い物リストにも含めない（越えてはならない制約）
- **FR-014**: System MUST 回避対象を除いて要求を満たせない場合、混入させる代わりにその旨と代替案を提示する
- **FR-015**: Users MUST be able to アレルギー・苦手食材を更新でき、更新内容は永続化されて以降の提案が常にそれを尊重する（在庫と異なり、毎回の再申告を要しない）

**対話での変更**

- **FR-016**: Users MUST be able to 提案された献立の変更・改善をチャットで依頼できる（例: 特定の日を別ジャンルに、もっと簡単に、辛さを抑える）
- **FR-017**: System MUST 変更を適用した際、献立と買い物リストを矛盾なく更新し、変更対象以外を不必要に変えない

**会話からの好み学習**

- **FR-018**: System MUST 会話の中でユーザーが述べた好み・味の感想を捕捉する。対象は食材の好き嫌いに加え、「辛い」「しょっぱい」「パンチが足りない」等の抽象的な味の属性を含む
- **FR-019**: System MUST 抽象的な味の感想を具体的な調整（例: 辛さを抑える、香辛料を足す、塩分を控える）に翻訳し、以降の提案に反映する
- **FR-020**: System MUST 特定レシピに対するフィードバックを、そのレシピ固有の改善として記憶し、次回同じ／類似のレシピを提案・生成する際に自動で適用する（例:「この麻婆豆腐はしょっぱい」→次回の麻婆豆腐は塩分を控える、「パンチが足りない」→にんにく等を足す）。料理に不慣れなユーザーが自力では言語化・実行しにくいレシピ改善を、エージェントが代行して積み上げる
- **FR-021**: System MUST 学習した好み（全体的な味の傾向・レシピ固有の調整の両方）を永続化し、セッションをまたいで以降の献立提案に反映する
- **FR-022**: System MUST 好みの蓄積が少ない段階でも破綻せず、無難な提案を返す
- **FR-023**: System MUST 味の感想が過去と相反する場合、直近の感想を優先して調整する
- **FR-024**: System MUST 好みの反映よりもアレルギー・苦手食材の回避（FR-013）を常に優先する

**会話全般**

- **FR-025**: System MUST 提案セッション内の会話文脈を保持し、続く発話を文脈に沿って解釈する
- **FR-026**: System MUST 曖昧・不完全な入力に対し、確認の質問をするか、合理的な前提を置いてその前提を明示する

### Key Entities _(include if feature involves data)_

- **回避制約 (DietaryConstraint)**: ユーザーに紐づくアレルギー・苦手食材の一覧。提案に対する絶対的な除外条件（ハード制約）。永続化され、セッションをまたいで保持される
- **好みプロファイル (PreferenceProfile)**: 会話から学習した好みの傾向（食材の好み、および辛さ・塩分・味の濃さ等の味の属性）。提案を寄せるためのソフトな優先で、回避制約より弱い。永続化される。会話で述べられた抽象的な味の感想（例:「しょっぱい」「パンチ不足」）は、解析して具体調整に翻訳したうえで本プロファイルに反映する（感想そのものを別途保存する実体は持たない）
- **在庫アイテム (InventoryItem)**: 手持ちの食材・調味料。品目と概算数量。当該セッション内でのみ保持し、セッションをまたいで永続化しない。個別の消費期限は持たず、一般的な日持ちはドメイン知識として扱う
- **献立 (MealPlan)**: 指定日数分の夕食の集合。対象期間を持つ
- **レシピ／食事 (Recipe/Meal)**: 献立を構成する個々の料理。必要な食材と分量、手順の要点
- **買い物リスト (ShoppingList)**: 献立成立に必要で手持ちにない食材の集合（品目と数量）
- **会話セッション (Conversation)**: 在庫把握・提案・変更のやり取りの文脈

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 登録済みのアレルギー・苦手食材が提案（献立・買い物リスト）に含まれる割合は 0%
- **SC-002**: 生成される買い物リストは、手持ちで不足するぶんの食材のみで構成される（手持ちで足りる食材を含めない）
- **SC-003**: ユーザーは在庫を伝え始めてから最初の献立提案を受け取るまで、通常の在庫量で3分以内に到達できる
- **SC-004**: 提案どおりに調理した場合、対象期間開始時の生鮮食材の80%以上が期間内に使用される計画になる
- **SC-005**: 指定日数が範囲外（0日以下・7日超）のリクエストは、100% が上限内に収める案内に至り、暴走した生成を発生させない
- **SC-006**: チャットで変更を依頼したユーザーの90%が、言い直しを繰り返さずに意図した変更を得られ、その際の買い物リストは献立と整合している
- **SC-007**: 抽象的な味の感想（例:「しょっぱい」「パンチが足りない」）を伝えたユーザーの90%が、具体的な調整（調味料・香辛料・味の濃さの調整）を得られる
- **SC-008**: 特定レシピに味の感想を伝えた場合、次回そのレシピが提案された 90% 以上で、該当する具体調整（例: 塩分・辛さ・香りの調整）が反映されている

## Assumptions

- 対象は単一ユーザー／単一世帯（MVP）。複数世帯や世帯内の共有・権限管理は範囲外
- 献立は1日あたり夕食1食のみを対象とする（MVP）。朝食・昼食、および複数食（3食）対応は範囲外
- 分量は既定の想定人数に基づく概算とする。世帯人数に応じた分量調整は本 spec の範囲外（後続 spec で扱う）
- 好み学習の入口は「会話でのフィードバック」に限定する。5段階評価インターフェースは本 spec の範囲外とし、同じ好み学習の核に後続 spec で足す
- 提案根拠の可視化、イラスト生成は本 spec の範囲外（後続 spec で扱う）
- レシピ保存は本 spec の範囲外（後続 spec で扱う）。将来的に、気に入ったレシピを保存でき、保存済みレシピを取り入れた献立を提案できるようにする想定
- 在庫は毎回チャットで申告する前提とし、セッションをまたいで記憶・持ち越ししない。使い切れなかった余りは次回の申告で伝える（毎週リセットする使い方を想定）。したがって「献立確定による在庫差し引き」「購入による在庫加算」「余剰の持ち越し」といった在庫ライフサイクルは扱わない
- 回避制約（アレルギー・苦手食材）と好みプロファイルは永続化され、次回以降のセッションでも利用できる（在庫と異なり、毎回の再申告を要しない）
- 回避制約はハード制約（絶対回避）、好みはソフトな優先であり、両者が競合する場合は回避を優先する
- 在庫ごとの消費期限は管理しない。一般的な日持ちのみ考慮し、冷凍などの保存判断はユーザーに委ねる
- Web アプリとして提供する。モバイル最適化は範囲外
- 高度な認証・アカウント基盤は本仕様の前提としない（必要性・方式は plan／ADR で検討）
- 自然言語の発話を在庫・要望として解釈し、制約下で献立を計画できる AI エージェント能力に依存する（具体的な技術・プロダクト選定は plan／ADR で扱う）
