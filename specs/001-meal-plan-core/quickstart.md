# Quickstart / 検証ガイド: 献立コア

本機能がエンドツーエンドで動くことを確認するための手順。実装コードは含めない（詳細は tasks.md と実装フェーズ）。

## 前提

- `pnpm install` 済み
- Gemini（Gemini Enterprise Agent Platform）へのアクセス資格情報を環境変数に設定
- ローカル DB（Cloud SQL Postgres 相当）に接続でき、Drizzle マイグレーション適用済み（`pnpm db:migrate`）

## 起動

```sh
pnpm dev   # http://localhost:3000
```

## 検証シナリオ

### S1. 在庫→献立・買い物リスト（US1）

1. チャットに「じゃがいも、人参、玉ねぎがある。3日分の夕食を考えて」と入力
2. 期待: 3日分の夕食献立と、手持ちで不足するぶんだけの買い物リストが返る
3. 期待: 手持ち食材が期間内で使い切られる構成／傷みやすい食材が早い日に配置される

### S2. アレルギー・苦手の絶対回避（US2）

1. 設定で「甲殻類アレルギー」「ピーマンが苦手」を登録
2. チャットで献立を要求
3. 期待: 献立・買い物リストのいずれにも甲殻類・ピーマンが一切含まれない（0%）
4. 手持ちにピーマンを申告 → 期待: 使い切り対象から除外され、献立に使われない

### S3. 対話での変更（US3）

1. 提案された献立の1日を指定し「この日を魚料理に」と依頼
2. 期待: 該当日だけ変更され、買い物リストが更新後の献立と整合する

### S4. 会話からの好み学習（US4）

1. 「麻婆豆腐は辛いから変えて」と依頼 → 期待: 該当が別レシピに変わり、以降の提案で辛い料理を避ける／辛さを抑える（全体傾向）
2. 特定レシピに「この生姜焼きは少ししょっぱかった」と伝える → 次回その生姜焼きが提案される → 期待: 塩分を控えた版で提案される（レシピ固有の改善）
3. 「パンチが足りない」と伝える → 期待: にんにく・香辛料を足す具体調整が反映される
4. 別セッションで再度献立を要求 → 期待: 学習した好み（全体傾向・レシピ固有調整）が反映される（永続）

## 自動テストでの検証（Vitest）

- 単体: 買い物リストが不足分のみ（INV-2）、日持ち順の配置、回避対象の突き合わせ検証
- 統合: エージェント経由で回避対象が 0% 混入（INV-1）、変更後の買い物リスト整合、好み注入の反映

```sh
pnpm test
```
