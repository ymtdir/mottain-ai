# Feature Specification: レシピカード（お気に入り登録・カード表示）

**Feature Branch**: `002-recipe-card`

**Created**: 2026-07-06

**Status**: Draft

**Input**: カレンダーに記録された料理をお気に入りとして登録し（US1）、登録したレシピをカード形式の一覧で見返し、材料・手順まで確認できる（US2）。料理のイラストは登録処理とは切り離してバックエンドで非同期に生成し、カードには生成状況（生成中・生成済み・生成失敗）を適切に表示する（US3）。001（献立コア）で範囲外とした「レシピ保存」を担う spec。チャット画面からの保存機能は廃止し、登録経路をカレンダーのみに統一した。

## Clarifications

### Session 2026-07-06

- Q: お気に入りは二値（保存/未保存）か、5段階評価つきか。評価つきなら好み学習に連動させるか → A: 二値のお気に入り（保存/解除のみ、評価なし）。純粋な「レシピ保存」に徹し、5段階評価・好み学習（001 で先送り）とは連動させない
- Q: 料理のイラストは本 spec の範囲か（生成のタイミングと失敗時の扱い） → A: 本 spec に含める。ただし登録処理とは非同期に分離し、登録は画像を待たず即時完了する。生成はバックエンドで行い、完了時にレシピへ紐付ける。カードでは生成中・生成済み・生成失敗の状態を画像欄に表示し、失敗時は再試行できる

## User Scenarios & Testing _(mandatory)_

### User Story 1 - カレンダーから料理をお気に入り登録する (Priority: P1)

ユーザーはカレンダーに記録された料理を「気に入った」と感じたとき、料理詳細ダイアログからお気に入りとして登録できる。登録すると、その料理の名前・材料・手順が保存され、あとから見返せるようになる。お気に入りは永続化され、セッションをまたいで残る。登録経路はカレンダーのみとし、チャット画面からの直接保存は提供しない。

**Why this priority**: 「実際に作って良かった料理を手元に残したい」という課題を解決する最小の価値。カレンダーに記録された料理（= 実際に承認・実食した料理）だけを保存対象にすることで、保存の意図が明確になる。

**Independent Test**: カレンダーに記録された料理の詳細ダイアログを開き、(a) 登録が受理され、(b) 別セッション（再読み込み後）でもそのレシピが保持されていることを確認できる。

**Acceptance Scenarios**:

1. **Given** カレンダーに料理が記録されている, **When** 詳細ダイアログから「お気に入りに登録」する, **Then** そのレシピが名前・材料・手順とともに保存される
2. **Given** レシピを登録済み, **When** ページを再読み込みする（別セッション）, **Then** 登録したレシピが保持されている
3. **Given** すでに登録済みのレシピ, **When** 同じレシピを再度登録しようとする, **Then** 重複して増えず、登録済みとして扱われる

---

### User Story 2 - 登録レシピをカード形式で一覧・閲覧する (Priority: P1)

ユーザーは登録済みのレシピを、カードが並ぶ一覧として見返せる。各カードには料理のイラスト欄とレシピ名が示される（イラストの生成状況は US3 で扱う）。カードを選ぶと、そのレシピの材料と手順を、チャットでの提案時と同じ形式で確認できる。

**Why this priority**: 保存（US1）した内容を実際に使う出口。保存と閲覧が揃って初めて「取っておいて後で作る」体験が完成するため、US1 と同格の P1。

**Independent Test**: 登録済みレシピが1件以上ある状態でお気に入り一覧を開き、(a) 各レシピがカードとして並び、(b) カードを選ぶと材料・手順が提案時と同じ形式で表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** レシピが登録されている, **When** お気に入り一覧を開く, **Then** 登録済みレシピがカードの一覧として表示され、各カードにレシピ名とイラスト枠が示される
2. **Given** カード一覧が表示されている, **When** 1つのカードを選ぶ, **Then** そのレシピの材料と手順が提案時と同じ形式で表示される
3. **Given** お気に入りが0件, **When** 一覧を開く, **Then** エラーにならず、空であることが分かる表示になる

---

### User Story 3 - 料理のイラストが非同期で用意される (Priority: P2)

ユーザーがレシピをお気に入り登録すると、登録自体は画像を待たずに即時完了する。料理のイラストは登録処理とは切り離してバックエンドで非同期に生成され、完成したレシピへ紐付けられる。生成が済むまでにカードを開いた場合でも、画像欄に現在の状態（生成中／生成失敗）が適切に示され、生成が済めばイラストが表示される。生成に失敗した場合は、その旨を示したうえで再試行できる。

**Why this priority**: イラストは体験を豊かにする重要な要素だが、生成には時間がかかる。保存・閲覧（P1）は画像がなくても成立するため、非同期の付加価値として P2。登録をブロックしない設計が肝心。

**Independent Test**: レシピを登録し、(a) 登録が画像生成を待たずに即時完了すること、(b) 生成前にカードを開くと「生成中」が表示されること、(c) 生成完了後はイラストが表示されること、(d) 生成失敗時は失敗状態が表示され再試行できること、を確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーがレシピを登録する, **When** 登録を実行する, **Then** イラスト生成の完了を待たずに登録が即時完了し、レシピは一覧に現れる
2. **Given** イラストがまだ生成中, **When** そのレシピのカードを開く, **Then** 画像欄に「生成中」を示す表示が出る
3. **Given** バックエンドでのイラスト生成が完了する, **When** そのレシピのカードを見る, **Then** 生成されたイラストが画像欄に表示される
4. **Given** イラスト生成が失敗した, **When** そのレシピのカードを開く, **Then** 画像欄に「生成失敗」を示す表示が出て、再試行できる
5. **Given** 生成に失敗したレシピ, **When** 再試行する, **Then** 生成が再度実行され、成功すればイラストが表示される

---

### User Story 4 - 登録レシピを削除する (Priority: P3)

ユーザーは不要になったレシピをお気に入りから削除できる。削除したレシピは一覧・カードから消え、以降は保持されない。

**Why this priority**: 一覧が増えたときの整理手段。保存・閲覧（P1）が成立していれば後付けできるため P3。

**Independent Test**: 登録済みレシピを削除し、一覧から消えること、再読み込み後も復活しないことを確認できる。

**Acceptance Scenarios**:

1. **Given** レシピが登録されている, **When** そのレシピを削除する, **Then** カード一覧から消える
2. **Given** レシピを削除した, **When** ページを再読み込みする, **Then** 削除したレシピは復活しない

---

### Edge Cases

- お気に入りが0件のとき（空状態を分かりやすく表示する）
- 同じレシピを重複して登録しようとしたとき（二重に増やさない）
- 手順や材料が欠けている／簡略なレシピ（例:「カレー（残り）」のように手順が空の日）を登録しようとしたとき
- 料理のイラストがまだ生成中のとき（画像欄に「生成中」を示す。登録やレシピ閲覧はブロックしない）
- 料理のイラスト生成に失敗したとき（画像欄に「生成失敗」を示し、再試行できる）
- 生成中にカードを開いたまま生成が完了したとき（状態が生成中→表示へ切り替わる）
- 登録件数が非常に多いとき（一覧がスクロール等で破綻しない）
- 生成途中・確定前のレシピを登録しようとしたとき（確定した内容のみを保存対象とする）

## Requirements _(mandatory)_

### Functional Requirements

**お気に入り登録**

- **FR-001**: Users MUST be able to チャットで提案・生成された個々のレシピ（献立の一品）をお気に入りとして登録できる
- **FR-002**: System MUST 登録時に、レシピ名・材料（品目と分量）・手順を、提案時と同じ形式・内容で保存する
- **FR-003**: System MUST 登録したお気に入りを永続化し、セッションをまたいで保持する（在庫と異なり毎回の再登録を要しない）
- **FR-004**: System MUST 同一レシピの重複登録を防ぐ（すでに登録済みのレシピは二重に増やさない）
- **FR-005**: System MUST お気に入りを二値（登録済み／未登録）として扱う。星評価・ランク付けは持たず、好み学習（PreferenceProfile）とも連動させない

**カード表示・閲覧**

- **FR-006**: System MUST 登録済みレシピをカード形式の一覧として表示する
- **FR-007**: System MUST 各カードにレシピ名と料理のイラスト欄を表示する（イラストの生成状況の扱いは FR-011〜FR-016 に従う）
- **FR-008**: Users MUST be able to カードから各レシピの材料と手順を、チャットでの提案時と同じ形式で確認できる
- **FR-009**: System MUST お気に入りが0件のとき、エラーにせず空状態を分かりやすく表示する

**削除**

- **FR-010**: Users MUST be able to 登録済みレシピをお気に入りから削除でき、削除は永続化される

**イラスト（非同期生成）**

- **FR-011**: System MUST お気に入り登録を、イラスト生成の完了を待たずに即時完了させる（登録は画像処理でブロックされない）
- **FR-012**: System MUST 登録後、料理のイラストをバックエンドで非同期に生成し、完了時に対象レシピへ紐付けて永続化する
- **FR-013**: System MUST 各レシピのイラスト生成状況（未生成／生成中／生成済み／生成失敗）を管理し、参照できるようにする
- **FR-014**: System MUST カードの画像欄に、生成状況に応じた表示を出す（生成中は「生成中」を示す表示、生成済みはイラスト、生成失敗は「生成失敗」を示す表示）
- **FR-015**: Users MUST be able to イラスト生成に失敗したレシピについて、生成を再試行できる
- **FR-016**: System MUST イラスト生成の成否にかかわらず、レシピの保存・一覧表示・材料/手順の閲覧を妨げない

### Key Entities _(include if feature involves data)_

- **保存レシピ (SavedRecipe)**: お気に入り登録された1つのレシピ。レシピ名、材料（品目と分量）、手順を、提案時と同じ内容で保持する。料理のイラストと、その生成状況（未生成／生成中／生成済み／生成失敗）を持つ。星評価は持たない（二値の登録済み状態のみ）。永続化され、セッションをまたいで保持される
- **イラスト生成ジョブ (IllustrationJob)**: 登録後に非同期でレシピのイラストを生成する処理単位。対象レシピと生成状況を持ち、成功時は生成物をレシピへ紐付け、失敗時は再試行できる。登録処理とは独立して動く
- **お気に入りコレクション (FavoriteCollection)**: ユーザーに紐づく保存レシピの集合。カード一覧の表示元。重複を持たない

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: ユーザーは提案されたレシピを、そのレシピを見ている画面から数アクション以内でお気に入り登録できる
- **SC-002**: 登録したレシピは、別セッション（再読み込み後）でも 100% カード一覧に表示される（永続化される）
- **SC-003**: カード一覧から任意のレシピの材料・手順に、1アクションで到達できる
- **SC-004**: 保存したレシピの名前・材料・手順は、提案時の内容と 100% 一致する（欠落・改変がない）
- **SC-005**: お気に入りが0件でもエラーが発生せず、空状態が表示される
- **SC-006**: 削除したレシピは、再読み込み後も 100% 復活しない
- **SC-007**: お気に入り登録は、イラスト生成の完了を待たずに即時完了する（登録操作が画像生成時間に引きずられない）
- **SC-008**: 生成中・生成失敗を含むどの生成状況でも、カードの画像欄に状況に応じた表示が 100% 出て、空欄や壊れた表示にならない
- **SC-009**: イラスト生成に失敗したレシピは再試行でき、再試行の成功時にはイラストが表示される

## Assumptions

- 対象は単一ユーザー／単一世帯（001 を踏襲）
- 保存対象は、チャットで提案・生成された献立中のレシピに限る。ユーザーが自由入力でレシピを手書き作成して登録することは本 spec の範囲外
- 料理のイラストは本 spec に含める。ただし登録処理とは非同期に分離し、登録は画像を待たず即時完了する。生成はバックエンドで行い、生成状況（未生成／生成中／生成済み／生成失敗）をカードの画像欄に表示する。イラスト生成の具体的な手段（生成モデル・保存先など）は plan／ADR で扱う
- お気に入りは二値（登録済み／未登録）とし、5段階の星評価・好み学習との連動は本 spec の範囲外。001 が「レシピ保存」と「5段階評価・好み学習」を別々の後続 spec に分けている分担を踏襲する
- 保存済みレシピを献立提案に取り込む（保存レシピを使った献立を提案する）ことは本 spec の範囲外。001 の想定（「保存済みレシピを取り入れた献立を提案できるようにする」）を踏襲し、後続 spec で扱う
- 「材料・手順は LLM 出力と同じ形式」とは、001 の献立提案でレシピが持つ構造（レシピ名・材料〔品目と分量〕・手順の要点）と同じ形式・粒度を指す
- Web アプリとして提供する。モバイル最適化は範囲外
- 高度な認証・アカウント基盤は本仕様の前提としない（必要性・方式は plan／ADR で検討）
