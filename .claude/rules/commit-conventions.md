# コミット規約

## プレフィックス

| プレフィックス | 説明                                                 | 例                                       |
| -------------- | ---------------------------------------------------- | ---------------------------------------- |
| `feat:`        | 新機能の追加                                         | feat: ユーザー認証を実装                 |
| `fix:`         | バグの修正                                           | fix: ログイン時のセッション保存エラーを修正 |
| `docs:`        | ドキュメントの変更                                   | docs: README にインストール手順を追加     |
| `style:`       | UI/UXの変更（見た目・デザインのみ）                  | style: ヘッダーのレイアウトを改善         |
| `refactor:`    | リファクタリング、パフォーマンス改善、テスト・CI基盤 | refactor: データベースクエリを最適化      |
| `test:`        | テストコードの追加・更新                             | test: ログイン機能の E2E テストを追加     |
| `chore:`       | 依存パッケージ更新、フォーマッタ整理、その他定型作業 | chore: 依存パッケージを更新               |

## コミットメッセージ形式

```
[プレフィックス]: [日本語で説明]
```

## 作業単位

**適切なコミット単位**:

- 関数を1つ追加する
- コンポーネントを1つ修正する
- テストケースを1つ追加する
- 定数や型定義を追加する
- インポートを整理する

**避けること**:

- 複数の機能をまとめた大きなコミット
- 「WIP」「仮」などのメッセージによるコミット
- 壊れた中途半端な状態でのコミット（下記TDD例外を除く）

**TDDレッドコミット例外**: テストファーストで開発する場合、実装前に失敗するテストをコミットしてよい。状態を明示的にメッセージに記載すること — 例: `test: validateEmail に空文字ケースを追加 (RED)`。次のコミットでは `fix:` または `feat:` を使ってグリーンにする。

## 作業単位の例

### バグ修正

```bash
fix: エラーメッセージ定数を定義
fix: バリデーション関数を修正
test: エッジケースのテストを追加
fix: コンポーネントに修正を適用
docs: 変更内容をコメントに記載
```

### 新機能

```bash
feat: 新しい API エンドポイントを定義
feat: データモデルを作成
feat: サービス層のロジックを実装
test: サービスのユニットテストを追加
feat: UI コンポーネントを作成
test: コンポーネントのテストを追加
docs: API 仕様を更新
```

### リファクタリング

```bash
refactor: 共通ロジックを抽出
refactor: 型定義を整理
refactor: 未使用コードを削除
test: リファクタリング後のテストを確認
chore: インポートを整理
```

## 注意事項

- **1コミット1目的**: 各コミットは明確な単一の目的を持つこと
- **具体的に**: 「fix」だけでなく、何を修正したかを明記する
- **作業単位**: 機能単位ではなく、実際の作業単位でコミットする
