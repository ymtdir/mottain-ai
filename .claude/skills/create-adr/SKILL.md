# create-adr

このプロジェクトの ADR（Architecture Decision Record）を、共通フォーマットで作成するためのスキル。
「なぜその技術を採用したか」「他にどんな候補があったか」を記録し、AI エージェントへのコンテキスト共有を目的とする。

検討した選択肢（代替案）も残せる、個人開発向けの軽量な ADR 形式を用いる。

## 進め方（draft-then-confirm）

引数で採用する技術・決定の対象を大まかに受け取り、**対話で内容を煮詰めてから**書き込む。
意思決定そのものはユーザーが行う。AI は選定を代行せず、叩き台の作成と抜け漏れチェックを担う。

1. **ドラフトを提示する**: 引数・直近の会話・リポジトリの文脈（ブランチ名、依存関係、
   コード等）から、テンプレートを 8 割方埋めたドラフトを提示する。
2. **核心の 3 点を確認する**: 以下に推測が混じる場合は、決めうちで書かず必ずユーザーに確認する。
   - 採用したもの（決定）
   - 採用理由
   - 不採用にした代替案と、その理由
3. **煮詰める**: ユーザーの修正・追記を反映し、合意できたら確定する。
4. **書き込む**: 「手順」に従ってファイルを作成する。

> ⚠️ 事実として確定できない理由や代替案を、それらしく捏造しない。
> 埋められない核心項目は空欄プレースホルダで残さず、短く質問して埋める。

## 手順

1. **保存先を確認する**: ADR は `docs/adr/` に置く。ディレクトリが無ければ作成する。

2. **連番を決める**: `docs/adr/` の既存ファイル（`NN-*.md`）を見て、最大番号 + 1 を
   ゼロ埋め 2 桁で採番する（例: 既存が `01`, `02` なら次は `03`）。最初の ADR は `01`。

3. **ファイル名を決める**: `NN-<スラッグ>.md` の形式。
   スラッグは決定の対象を表す短い kebab-case（英小文字）。
   例: `01-<対象>.md`, `02-<対象>.md`。

4. **内容を埋める**: 下記テンプレートに沿って書く。ユーザーとの会話や、
   すでに決まっている技術選定の背景から埋められる部分は埋める。
   不明な項目は空欄プレースホルダのまま残さず、ユーザーに簡潔に確認する。

5. **本文の言語**: 会話の言語に合わせる（このプロジェクトでは日本語）。

## 書き方の指針

- **長さ**: 個人開発として十分な密度で、冗長にしない。各セクション数行〜十数行が目安。
- **具体的に**: 「モダンだから」ではなく「型安全 / 学習コスト / エコシステム」など
  判断軸を明示する。将来の自分と AI が読んで判断を再現できるレベルで書く。
- **代替案は必ず 2 つ以上**書く。なぜ不採用にしたかまで書くことに価値がある。
- **status** は新規作成時は必ず `Accepted`。値の一覧と使い分けは「ステータスと置き換え」を参照。
- **date** は今日の日付（YYYY-MM-DD）。
- **段落内で手動改行しない**（1 段落 1 行）。折り返しはエディタ／ビューアに任せる。

## ステータスと置き換え

status は次の 3 つだけを使う。

| status | 意味 |
|---|---|
| `Accepted` | 採用中（現行の決定） |
| `Superseded by [NN-slug](NN-slug.md)` | 別の ADR に置き換えられた。本文は残す |
| `Deprecated` | 後継なしで廃止・使わなくなった |

技術を乗り換えるなど、既存の決定を**置き換える**とき:

1. 新しい ADR を通常どおり `Accepted` で作成し、本文に旧 ADR へのリンクと
   「`NN-旧` を置き換える」旨を書く。
2. 旧 ADR は削除せず、status だけ `Superseded by [NN-新](NN-新.md)` に変更する。

## テンプレート

テンプレートは [`assets/template.md`](assets/template.md) にある。
これを読み込んで内容を埋め、`docs/adr/NN-<スラッグ>.md` として書き出す。

## 完了後

作成したファイルのパスをユーザーに伝える。
既に多数の ADR がある場合や `docs/adr/README.md` が存在する場合は、
索引への 1 行追記も提案する（存在しなければ提案不要）。
