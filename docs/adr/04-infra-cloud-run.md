# 04. インフラ基盤に Cloud Run + Cloud SQL を採用し Terraform で管理する

- status: Accepted
- date: 2026-07-02

## コンテキストと課題

本ハッカソン（DevOps × AI エージェント）は応募要件として Google Cloud のアプリケーション実行プロダクトを 1 つ以上使うことを必須としており、ホスティング先は GCP から選ぶ前提が確定している。その中で、アプリケーションの実行環境・データ永続化・インフラ管理方法・デプロイ方法を、個人開発かつハッカソンの時間制約下で軽量に、かつ AI エージェントが追従できる形で決めたい。特に「何を作るか」が固まる前でも、後戻りの小さい土台（実行基盤と CD）は先に立てて、以降は push するたびに自動デプロイされる状態にしたい。

## 検討した選択肢

GCP 内の実行プロダクトに絞って比較する（他クラウドや自前サーバーは要件外のため検討しない）。

- Cloud Run（採用）
- Google Kubernetes Engine (GKE)
- Google Compute Engine (GCE)

## 決定

「アプリの実行基盤に Cloud Run、データベースに Cloud SQL for PostgreSQL を採用する。インフラは Terraform で IaC 管理し、デプロイは GitHub Actions から Cloud Run へ行う CD を組む。」

- 実行環境: **Cloud Run**（コンテナをデプロイ、ゼロスケール）
- データベース: **Cloud SQL for PostgreSQL**（基本構成として採用。JSON 形式で扱いたいデータは PostgreSQL の JSONB 型で持つ。実際のスキーマ・接続はアプリ仕様確定後に足す）
- IaC: **Terraform**（構成をコード化し、state はリモートバックエンドで管理）
- CD: **GitHub Actions** → Cloud Run への自動デプロイ

上記以外の GCP プロダクト（キャッシュ、ストレージ、キュー等）は必要になった時点で後続 ADR または本 ADR の更新で足す（YAGNI）。

### 理由

- Cloud Run はコンテナを渡すだけで動き、ゼロスケールで待機コストがかからない。個人ハッカソン規模の Web/API 用途に最も手離れが良い。
- GKE / GCE のようなクラスタ運用・OS 管理が不要で、DevOps の手数を実装に回せる。
- Cloud SQL はマネージドな RDB で、慣れた SQL をそのまま使える。エンジンは PostgreSQL を選ぶ。リレーショナルに加え JSONB 型で JSON データも扱え、要件を賄える。MySQL・SQL Server を積極的に選ぶ理由が現状ないため、機能面で無難な PostgreSQL に寄せる。
- Terraform で構成を残すことで、AI エージェントがインフラの現状をリポジトリから読めて、再現・変更もレビュー可能になる。
- CD を最初に通すことで、以降の機能追加が push だけでデプロイされ、イテレーションが速くなる。DevOps 基盤の整備自体がハッカソンの成果物価値でもある。

## 各選択肢の比較

### Cloud Run（採用）

- 👍 良い点: コンテナを渡すだけで動く / ゼロスケールで低コスト / インフラ運用がほぼ不要 / Terraform・GitHub Actions との連携が容易
- 👎 懸念点: 長時間実行や常駐処理・きめ細かいネットワーク制御には不向き

### Google Kubernetes Engine (GKE)

- 👍 良い点: 複数サービス・複雑なオーケストレーションに強く、拡張性が高い
- 👎 不採用の理由: クラスタの構築・運用コストが個人ハッカソンには過剰。要件に対して重量級（YAGNI に反する）

### Google Compute Engine (GCE)

- 👍 良い点: VM を自由に構成でき、任意のミドルウェアを載せられる
- 👎 不採用の理由: OS・パッチ・スケーリングを自前で管理する必要があり手数が多い。サーバーレスで足りる要件にはオーバーヘッドが大きい

## 結果

- アプリを Cloud Run 上で動かし、push からデプロイまでを自動化した状態で開発を始められる。
- インフラ構成が Terraform としてリポジトリに残り、AI も現状を参照・変更できる。
- データは Cloud SQL for PostgreSQL に置き、JSON は JSONB 型で扱う。
- トレードオフとして、Cloud Run の制約（実行時間・常駐不可など）に収まる設計を選ぶ必要がある。
- DB の詳細（インスタンス構成・スキーマ）や追加プロダクトは、アプリ仕様の確定後に足す。ドキュメント指向やリアルタイム同期が必要になった場合は Firestore の導入を後続 ADR で検討する。想定を超える実行要件が出た場合は GKE 等への移行も後続 ADR で再検討する。
