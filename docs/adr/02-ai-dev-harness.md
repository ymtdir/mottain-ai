# 02. 開発フローに自作ハーネス github-toolkit を採用する

- status: Accepted
- date: 2026-07-02

## コンテキストと課題

本プロジェクトは AI エージェント（Claude Code）を主体に開発を進める。その際、Issue 起票・ブランチ運用・コミット・PR 作成といった定型フローを、エージェントが毎回ブレなく実行できることが重要になる。素の状態ではフローが会話ごとに揺れ、コミット粒度やメッセージ、PR の書式も一定しない。個人開発かつハッカソンの時間制約下で、軽量かつ一貫した開発フローを、AI が自動で追従できる形で用意したい。

なお github-toolkit は、新機能開発に用いる Spec Kit（ADR 03）とは役割が異なり、バグ修正・UI 改善・リファクタリング・小さな機能追加など、仕様を立てる必要のない単発の作業を担う。

なお、本ハッカソンのテーマ自体が「DevOps × AI エージェント」であり、開発フローをエージェント向けに整備すること自体が成果物の一部でもある。

## 検討した選択肢

- 自作ハーネス [github-toolkit](https://github.com/ymtdir/github-toolkit)（採用）
- 既製の Claude Code 拡張フレームワーク（SuperClaude / BMAD-METHOD / claude-flow など）
- GitHub ネイティブ機能 + 素の Claude Code（ベースライン）

## 決定

「開発フローの標準化に、自作の Claude Code スキル群 [github-toolkit](https://github.com/ymtdir/github-toolkit)（`create-issue` / `create-pr` / `resolve-issue` + コミット規約）を採用する。あわせて本リポジトリ独自の `create-adr` スキルを併用する。各スキルは symlink ではなく実ファイルとして `.claude/` に取り込み、git 管理下に置く。」

詳細な各スキルの挙動は各 `SKILL.md`（`.claude/skills/*/SKILL.md`）およびコミット規約（`.claude/rules/commit-conventions.md`）を参照。

### 理由

- 自作なので中身を完全に把握でき、個人開発向けに機能を削って軽量化・改変できる。（実際、導入時に過剰だったラベル分類エンジン等を削って本プロジェクト向けに軽くした）
- 有名フレームワーク（SuperClaude / BMAD-METHOD 等）は多機能だが重量級で、必要なのは開発方法論ではなく Issue/PR/コミットの軽量な定型化だった。
- draft-then-confirm 方式に統一され、`create-adr` を含む全スキルで操作感が一貫する。
- コミット規約や Issue/PR の書式をスキルに埋め込むことで、AI が明示指示なしに規約へ追従する。
- copy 方式で `.claude/` に実ファイルとして入るため、clone・移動しても壊れず、差分がレビュー可能で AI もリポジトリ内から直接参照できる。

## 各選択肢の比較

### 自作ハーネス github-toolkit（採用）

- 👍 良い点: 完全に自分で制御・改変できる / 運用に合わせて軽量化済み / draft-then-confirm で一貫 / 実ファイルで可搬・AI が参照しやすい
- 👎 懸念点: メンテナンスを自分で負う / 既製ツールほど機能は多くない

### 既製の Claude Code 拡張フレームワーク（SuperClaude / BMAD-METHOD / claude-flow など）

- 👍 良い点: 多機能で実績が豊富（いずれも 2〜5 万 stars）、導入が速くメンテを他者に任せられる
- 👎 不採用の理由: 独自の方法論・役割・多エージェントを持ち込む重量級。個人ハッカソンには過剰で、軽量な運用に合わせて削るより自作した方が早い

### GitHub ネイティブ機能 + 素の Claude Code（ベースライン）

- 👍 良い点: 追加物ゼロ。Issue/PR テンプレや Actions は標準機能で完結
- 👎 不採用の理由: AI が従うべきフローが明文化されず、会話ごとに操作がブレる。規約を毎回指示する手間が残る

## 結果

- Issue・PR・Issue 解決・ADR 作成が、スキルを通じて一貫した手順で実行できる。
- コミット規約が AI に自動追従され、履歴の粒度と書式が安定する。
- トレードオフとして、ハーネス自体の改善・追従は自分で行う必要がある。
- ハーネスの構造や思想を変えるときは、本 ADR を更新するか後続 ADR で置き換える。
