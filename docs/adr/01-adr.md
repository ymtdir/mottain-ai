# 01. 意思決定の記録に ADR を採用する

- status: Accepted
- date: 2026-07-02

## コンテキストと課題

個人開発かつ AI エージェントを積極的に活用して開発を進める。技術選定やアーキテクチャ上の判断が積み重なるが、「なぜその選択をしたか」「他に何を検討したか」が残らないと、後から自分も AI も判断を再現・再利用できない。特に AI エージェントへコンテキストを渡す際、決定の背景が構造化・テキスト化されていると効果が高い。個人開発でも無理なく続けられる、軽量な記録方式が欲しい。

## 検討した選択肢

- ADR（採用）
- 記録しない / コミットメッセージのみ
- 外部サービス（Notion などのドキュメントツール）
- GitHub Issues / Discussions

## 決定

「意思決定の記録に ADR を採用する。1 決定 1 ファイルで `docs/adr/NN-<スラッグ>.md` に蓄積し、検討した選択肢も併せて残す構成とする。作成は create-adr スキルで統一する。」

### 理由

- Markdown でリポジトリ内に置くため、コードと同じく Git でバージョン管理でき、変更履歴と紐づく。
- プレーンテキストなので AI エージェントが読み込みやすく、コンテキスト共有に向く。
- 検討した選択肢（代替案）とその不採用理由を構造的に残せる。
- 1 決定 1 ファイル・数十行の軽量運用なので、個人でも継続できる。

### ステータスの運用

status は次の 3 つだけを使う。

- `Accepted`: 採用中（現行の決定）
- `Superseded by [NN-slug](NN-slug.md)`: 別の ADR に置き換えられた。本文は残す
- `Deprecated`: 後継なしで廃止・使わなくなった

一般的な ADR にある `Proposed` と `Rejected` は使わない。

- `Proposed`（提案中）を使わない理由: 個人開発で提案段階の合議がなく、採用を決めた時点で書くため、「提案中」の状態が発生しない。
- `Rejected`（却下）を使わない理由: 不採用にした案は、採用した ADR の「検討した選択肢」に不採用理由付きで残す。却下だけの独立した ADR は作らない。

## 各選択肢の比較

### ADR（採用）

- 👍 良い点: リポジトリ内で Git 管理 / AI・人が読みやすい / 代替案を残せる / 軽量で継続しやすい / 広く使われた形式
- 👎 懸念点: 決定ごとに手で書く規律が必要

### 記録しない / コミットメッセージのみ

- 👍 良い点: 追加の手間がゼロ
- 👎 不採用の理由: 理由・代替案が残らず、後から判断を追えない。AI にも渡せない

### 外部サービス（Notion などのドキュメントツール）

- 👍 良い点: 編集・閲覧が手軽
- 👎 不採用の理由: コード／Git から分離し、AI エージェントやレビューの導線から外れる

### GitHub Issues / Discussions

- 👍 良い点: リポジトリに紐づく / 議論の経緯も残せる / 追加ツール不要
- 👎 不採用の理由: 決定が複数の issue に散らばり一覧しにくい / コードと一緒にチェックアウト・レビューされない / AI に渡すとき集めるのが手間

## 結果

- 今後の技術・アーキテクチャ上の決定は `docs/adr/` に ADR として蓄積する。
- create-adr スキルにより形式が統一され、AI へのコンテキスト共有が改善する。
- トレードオフとして、決定のたびに ADR を書く手間が発生する。
- 数が増えたら `docs/adr/README.md`（索引）の追加を検討する。
