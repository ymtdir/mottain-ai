# 14. 食事カレンダーの記録に「スナップショット保持・お気に入りと非統合」の meal_logs を採用する

- status: Accepted
- date: 2026-07-07

## コンテキストと課題

feature 003（食事カレンダー）では、チャットで提案・承認された献立を「その日に食べた料理」として記録し、月単位のカレンダーで見返す（FR-001〜011）。ここで 2 つの設計判断が要る:「①食べた料理をどのデータモデルで保存するか」と「②記録がどの契機で作られるか」。

前提として、002 の `saved_recipes`（お気に入り＋イラスト）は既に schema 定義があるが、マイグレーションは未適用で実データはゼロである。そのため「002 を作り替えて全レシピ＋お気に入りフラグに統合する」といったスキーマ刷新も選択肢に入りうる。一方で仕様は「細かな違いのある同名料理は毎回別記録として残す（重複排除しない）」「カレンダーにイラストは出さない（イラストはお気に入り時のみ・002 のまま）」を求めており、単一ユーザー・ハッカソン規模で新インフラ（憲章 I）を増やさない方針も効く。

## 検討した選択肢

食事記録のデータモデル:

- meal_logs にスナップショット保持し、お気に入り（saved_recipes）と分離する（採用）
- 正準レシピ＋isFavorite フラグへ 002 スキーマを刷新し、カレンダーは recipeId で参照する
- 埋め込み・ベクトル検索で同名料理を類似判定して正準化・重複排除する

記録の契機:

- エージェントが承認を判断して `recordMealPlan` ツールを呼んだときだけ記録する（採用）
- 明示的な確認 UI（「カレンダーに追加しますか？」）で記録する
- チャット完了後に会話全体を事後解析して自動抽出する

## 決定

食事記録は次の方式を採用する。

**データモデル**: 新テーブル `meal_logs` を追加し、日付（`eatenOn`）＋料理内容（`content` に title/ingredients/steps を JSONB でスナップショット）を自己完結で保持する。重複排除はしない（同名・同日でも別レコード）。002 の `saved_recipes` は変更せず、カレンダーは独立させる。将来お気に入りへ紐付ける場合に備え nullable な `sourceRecipeId` を後から足す余地は残すが、本 spec では持たない。カレンダーからのお気に入り登録は 002 の `registerRecipe`／`POST /api/recipes` をそのまま再利用する。

**記録の契機**: 承認の検知はエージェントのツール呼び出しをゲートにする。`runAgent` の tools に `recordMealPlan` を1つ追加し、ユーザーが提案献立を承認したとモデルが判断したときだけ記録する（自由会話文からの曖昧な自動記録はしない）。これは [ADR-07](07-agent-framework-vercel-ai-sdk.md) のエージェント基盤の適用であり、新規の枠組みは足さない。日付は承認日を起点に `eatenOn = 承認日 +（Recipe.day − 1）`（Asia/Tokyo 基準）で決定論的に割り当てる。

### 理由

- スナップショット保持なら join も id 紐付けも不要で最も単純。「細かな違いは別記録」の仕様と自然に一致し、レシピ編集が過去のカレンダーを書き換える副作用も起きない。
- 002 のスキーマを触らないため、別ブランチで行うマイグレーション作業と衝突しない。実データがゼロでも、あえて作り替えるだけの利得がない。
- 記録契機をツール呼び出しに閉じ込めると、記録の契機が明示的になり誤記録・二重記録を防げる（FR-002 / SC-006）。承認判断をエージェントに委ねることは憲章 V（AI が価値の中心）にも合致する。
- いずれも新 GCP インフラ（pgvector・キュー等）を増やさない（憲章 I）。

## 各選択肢の比較

### meal_logs にスナップショット保持・お気に入りと分離（採用・データモデル）

- 👍 良い点: 最も単純（join/参照なし）／「細かな違いは別記録」と整合／002 を触らずマイグレーション作業と非衝突／新インフラ不要
- 👎 懸念点: 同じ料理を複数回食べると内容が重複して保存される／「この料理を過去いつ食べたか」の集約は別途クエリが要る（現規模では許容）

### 正準レシピ＋isFavorite フラグへ 002 を刷新（不採用・データモデル）

- 👍 良い点: レシピが 1 行に集約され、カレンダーは recipeId 参照で軽い／お気に入りと記録が単一表で扱える
- 👎 不採用の理由: 002 スキーマの作り替えが必要／dedup が「細かな違いは別記録」方針と衝突／レシピ編集が過去のカレンダー表示を書き換える副作用

### 埋め込み・ベクトル検索で類似判定・正準化（不採用・データモデル）

- 👍 良い点: 表記ゆれを吸収して同一料理をまとめられる
- 👎 不採用の理由: pgvector 等の新インフラが必要で現規模には過剰（YAGNI・憲章 I）。**必要になれば導入する upgrade path** とする

### エージェントのツール呼び出しをゲートに記録（採用・契機）

- 👍 良い点: 記録契機が明示的で誤記録を防げる／既存ツール基盤（ADR-07）の適用で新規枠組み不要／エージェント中心（憲章 V）
- 👎 懸念点: 承認検知はモデル判断に依存する（取りこぼし・誤検知はありうる）→ 誤記録は削除（FR-012）で正す

### 明示的な確認 UI／会話全体の事後解析（不採用・契機）

- 👍 良い点: 確認 UI は誤検知が減る／事後解析はクライアント発火に依存しない
- 👎 不採用の理由: 確認 UI は「承認だけで反映」（SC-002）の体験を損なう。事後解析は誤記録・二重記録の温床

## 結果

- 新 GCP インフラを一切増やさずに、食事記録・カレンダー閲覧・カレンダーからのお気に入り登録を実装できる。002 のスキーマを触らないため、別ブランチのマイグレーション作業と独立して進められる。
- データモデル（`meal_logs`）・日付割当・ツール契約は [specs/003-meal-calendar/data-model.md](../../specs/003-meal-calendar/data-model.md) を正とし、記録・当月抽出・日付割当は `src/server/services/meal-log.ts` に、承認ツールは `src/server/agent/tools/record-meal-plan.ts` に実装する。
- トレードオフと将来: 「お気に入りをいつ食べたか辿る」需要が出たら `meal_logs.sourceRecipeId` でお気に入りへ紐付ける。004（コメント起点の好み学習）は料理名をキーに 001 の PreferenceProfile（recipeAdjustments）へ載せ、`meal_logs` には評価・コメントを持たせない。明示的な日付指定・類似の正準化は将来の拡張余地とする。
