# 03. 仕様駆動開発に GitHub Spec Kit を採用する

- status: Accepted
- date: 2026-07-02

## コンテキストと課題

AI エージェント（Claude Code）主体で開発を進めるが、思いつきのプロンプトから実装させると、要件が曖昧なまま進み手戻りが増える。特にハッカソン規模でも、何を作るか（仕様）→どう作るか（計画）→具体作業（タスク）を段階的に固めてから実装に入る流れを、AI が追従できる形で用意したい。ADR 02 で導入した github-toolkit は git/Issue/PR の運用を担うが、その前段の「仕様から実装への落とし込み」を型化する仕組みが別途必要になる。

なお、本ハッカソンのテーマは「DevOps × AI エージェント」であり、仕様駆動開発（Spec-Driven Development）を実践すること自体が成果物の一部でもある。加えて、注目度の高い Spec Kit と仕様駆動開発そのものを学ぶことも、今回の採用目的に含まれる。

## 検討した選択肢

- GitHub Spec Kit（採用）
- 自作の仕様駆動フロー（Markdown で spec/plan/tasks を自前運用）
- BMAD-METHOD（アジャイル多エージェント手法）
- 仕様を持たずアドホックにプロンプトで実装（ベースライン）

## 決定

「仕様駆動開発の基盤として GitHub Spec Kit を採用する。`specify` CLI で本リポジトリに初期化し、仕様→計画→タスク→実装のフェーズを Claude Code のスラッシュコマンドで進める。生成される spec / plan / tasks はリポジトリに含めて git 管理する。」

github-toolkit（ADR 02）とは補完関係で、Spec Kit が計画〜実装、github-toolkit が Issue/PR/コミット運用を担う。

### 理由

- 仕様→計画→タスク→実装の各フェーズが型化され、AI が段階的に、要件を確定させながら実装へ進める。
- GitHub 公式かつ巨大なコミュニティ（~117k stars）で、実績と継続的な更新が見込める。
- 特定エージェントに依存せず Claude Code に対応しており、既存のワークフローに載せられる。
- 生成物がリポジトリ内の Markdown なので、ADR と同様に AI がコンテキストとして参照できる。
- 学習目的。注目度の高い Spec Kit と仕様駆動開発の実践知を得ること自体を目的の一つとする。

## 各選択肢の比較

### GitHub Spec Kit（採用）

- 👍 良い点: 仕様駆動のフェーズが CLI/コマンドで型化 / GitHub 公式・~117k stars で実績十分 / Claude Code 対応 / 生成物が git 管理可能な Markdown
- 👎 懸念点: 手順とドキュメント（spec/plan/tasks）の管理コストが増える / 小さな変更にはオーバーヘッド

### 自作の仕様駆動フロー

- 👍 良い点: 完全に自分の運用に合わせられ、軽量にできる
- 👎 不採用の理由: 実績あるツールがある中で車輪の再発明になる。フェーズ設計やコマンド整備を自前で保守する負担が大きい

### BMAD-METHOD（アジャイル多エージェント手法）

- 👍 良い点: PM/Architect 等の役割を持つ本格的なアジャイル AI 開発手法
- 👎 不採用の理由: 多エージェント・プロセスが前提で個人ハッカソンには重い（ADR 02 と同じ理由）

### アドホックにプロンプトで実装（ベースライン）

- 👍 良い点: 準備ゼロですぐ書き始められる
- 👎 不採用の理由: 要件が曖昧なまま進み手戻りが増える。仕様が残らず AI にも意図が伝わりにくい

## 結果

- 仕様→計画→タスク→実装のフェーズに沿って、AI 主体の開発を進められる。
- spec / plan / tasks がリポジトリに残り、意図と経緯が追える。
- トレードオフとして、フローとドキュメントの管理コストが増える。
- 運用が重すぎると感じた場合は、フェーズの一部を省略するなど運用を見直すか、後続 ADR で方針を更新する。
