# 12. GitHub 上のコードレビューに CodeRabbit を採用する

- status: Accepted
- date: 2026-07-03

## コンテキストと課題

コードレビューを2層に分けて運用したい。エディタ内・実装中のインラインレビューは Claude Code の skills で行う（作業の流れを止めない）。一方で、PR 単位の GitHub 上でのレビューをどう担うかを決めたい。個人開発のためレビュアーは実質自分ひとりで、客観的なレビューの目が不足しがちという課題がある。候補は、GitHub でも自分の Claude Code を使って PR をレビューする案と、外部サービス CodeRabbit に委譲する案。

## 検討した選択肢

- CodeRabbit（採用）
- GitHub Copilot code review
- Qodo Merge / PR-Agent（オープンソース版あり）
- Greptile
- 自分の Claude Code で PR レビューも行う

> 注記: 各サービスの無料枠・価格は変動が速い。以下は 2026-07 時点で確認した内容にもとづく。導入時に最新の提供条件を確認すること。

## 決定

GitHub 上（PR 単位）のコードレビューを CodeRabbit に委譲する。インライン・実装中のレビューは従来どおり Claude Code の skills で行い、役割を分担する。

### 理由

- public リポジトリに対しては Pro 相当の機能を無期限で無料提供しており（レートリミットは 200 ファイル/時・4 レビュー/時程度）、比較した中で OSS への無料枠が最も手厚い。個人開発の public リポジトリという本プロジェクトの前提と噛み合う。自分の Claude Code でレビューすると利用枠を消費するため、CodeRabbit に委譲することで実装や設計の対話に枠を回せる。
- 別系統のレビュアーを挟むことで視点の多様性が生まれる。実装に使っている Claude Code と同じモデル・同じ文脈でレビューすると盲点を共有しやすいが、独立した CodeRabbit を通すことでそれを補える。
- PR 作成をトリガーに自動でレビューが走るため、レビュー漏れが起きにくい。手動で「レビューして」と依頼する運用より確実。
- サマリ・walkthrough を自動生成し、レビューコメントを PR 上に残す。レビュー履歴が GitHub に蓄積され、後から追える（[開発憲章](../../.specify/memory/constitution.md) 原則 IV「AI が追従できるコンテキスト」とも整合）。
- セットアップは GitHub App の導入のみで、CI やワークフローの追加実装が不要。個人開発の軽さ（原則 I）を保てる。

## 各選択肢の比較

### CodeRabbit（採用）

- 👍 良い点: public リポジトリに Pro 相当を無期限無料提供で、OSS 向け無料枠が比較中で最も手厚い / 自前の Claude Code 枠を節約できる / 独立したレビュアーで視点が多様になる / PR 作成で自動起動しレビュー漏れを防ぐ / サマリと履歴が PR に残る / GitHub App 導入のみで CI 実装が不要 / ノイズの少なさに定評
- 👎 懸念点: 外部サービス依存（無料枠・提供条件の変更リスク） / private 化した場合は有料（$24〜/dev/月程度） / レビュー品質はサービス側の実装に依存する

### GitHub Copilot code review

- 👍 良い点: GitHub ネイティブで統合が最もスムーズ / Copilot 契約があれば追加サービス不要
- 👎 不採用の理由: レビューが他の Copilot 機能と共通の premium request 枠を消費し、public リポジトリ向けの無制限無料枠がない / コンテキストが PR の diff に限られ、リポジトリ全体を見ないためクロスファイルの不具合を取りこぼしやすい

### Qodo Merge / PR-Agent（オープンソース版あり）

- 👍 良い点: PR-Agent としてオープンソースの自己ホスト版があり無料で動かせる / レビューとテスト自動生成を兼ねる / 主要 4 ホストに対応
- 👎 不採用の理由: 自己ホスト版は自前で LLM の API キー・コストを用意する必要があり「枠の節約」という狙いに合わない / ホスト版は有料（$30/user/月程度）で public 無料枠の手厚さは CodeRabbit に劣る / セットアップが重く個人開発の軽さ（原則 I）に反する

### Greptile

- 👍 良い点: リポジトリ全体をインデックスして文脈を踏まえたレビューを行い、バグ検出率が高いとされる
- 👎 不採用の理由: 有料前提（$30/dev/月程度・レビュー数上限あり）で、本プロジェクトが重視する public リポジトリの無料枠という軸に合わない

### 自分の Claude Code で PR レビューも行う

- 👍 良い点: 追加の外部サービスが不要 / プロジェクト文脈を深く共有した状態でレビューできる
- 👎 不採用の理由: 実装に使う利用枠をレビューでも消費する / 実装と同じモデル・文脈のため盲点を共有しやすい / PR ごとに手動起動が必要でレビュー漏れが起きやすい

## 結果

- インラインレビューは Claude Code skills、PR レビューは CodeRabbit という2層体制になり、それぞれの得意領域で自動レビューが効く。
- 自前の Claude Code 利用枠を実装・設計に集中させられる。
- トレードオフ: 外部サービスへの依存が増える。無料枠の縮小や private 化の際は運用の見直しが必要になる。
- 関連するフォロー: CodeRabbit の GitHub App を対象リポジトリに導入する。必要に応じて `.coderabbit.yaml` でレビュー方針（言語・除外パスなど）を調整する。
